# B端PRD质量检查 Prompt

> 作者：杨堃，《决胜B端》作者，B端产品专家
> 基于：《决胜B端（第二版）》《决胜体验设计》《决胜B端PRD模板 v2.0》
> 本 Prompt 用于对 B端产品的 PRD 和系统设计方案做全面、系统化的质量检查，覆盖 14 个维度。

## 使用说明

1. 先把待检查的 PRD / 系统设计文档内容完整提供给模型，或上传文档
2. 再发送本 Prompt
3. 模型需要先完成产品定型，再按下述顺序逐维度检查，每个维度检查完立即输出详细分析，最后输出汇总报告

适用场景：

- PRD 评审前自查
- 设计方案质量回顾
- 新人 PRD 写作指导
- 产品方案完整性验证

## 第零步：产品定型

在开始任何维度检查前，先阅读文档并判断以下两个维度，记录结论供后续检查使用：

**A. 商业属性**

- 自研内部系统：企业内部使用，不对外销售，如内部 OA、ERP、CRM
- 商业化产品：对外销售或运营，存在付费客户或商业变现，如 SaaS、平台产品

**B. 功能类型**

- 业务型管理软件：支撑核心业务流程运转，有复杂业务对象和状态管理
- 工具型软件：提供独立功能提升效率，无复杂业务对象关系
- 交易型平台：撮合多方完成交易
- 基础服务型：提供底层能力供其他系统调用

**根据定型结论，以下维度按条件适用：**

| 维度 | 适用条件 | 不适用时处理方式 |
| --- | --- | --- |
| 07-数据建模（ER图） | 仅适用于业务型管理软件和交易型平台 | 工具型或基础服务型标注“不适用”，仅检查关键数据结构 |
| 10-商业分析 | 仅适用于商业化产品 | 自研内部系统仅执行投入产出分析 |
| 03 维度中的竞品分析 | 仅适用于商业化产品 | 自研内部系统改为同类系统参考 |
| 06-企业架构层 6.5-6.9 | 仅适用于 0-1 新系统设计或系统级规划 PRD | 迭代需求或模块级需求只做 6.1-6.4 |
| R8-多租户风险项 | 仅适用于商业化 SaaS 产品 | 非 SaaS 标注“不适用” |
| 13-AI 功能 | 仅在文档涉及 AI 功能时评估 | 否则标注“不适用” |

## 检查输出规范

**核心原则：逐维度即时输出。**

每完成一个维度检查，就立即输出该维度的详细分析，然后再继续下一个维度。禁止把所有维度攒到最后统一输出。

**每个维度使用以下格式：**

```md
## 维度[编号] - [名称] ｜ 评级：[优秀 / 合格 / 待改进 / 严重缺失]

### 具体发现

**发现 1：[问题标题]** [P0/P1/P2/P3]
- PRD定位：第X节/[功能名称]
- 问题描述：[具体说明问题，不要泛泛而谈]
- 改进示例：[给出可立刻执行的改法]

### 隐性问题推断
结合产品类型和业务场景，列出 PRD 没写但按道理必须考虑的问题。
```

**最低输出要求：**

- 每个维度至少输出 3 条具体发现，若确实无问题，也要逐条说明理由
- 每条发现必须引用 PRD 中的具体章节、流程、功能名或界面名
- 每条发现必须给出具体可执行的改进示例
- 维度 09 必须逐个检查文档中出现的界面、弹窗、表单、操作，并指出缺失的交互说明

# 业务分析质量检查

> Supporting reference for the `check-prd` skill. 在执行业务分析质量检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

业务分析是B端产品设计的根基。《决胜B端》明确指出：B端产品建设的第一步是"市场分析与业务调研"，产品经理必须在充分理解业务的基础上才能进行方案设计。业务分析不扎实，后续所有设计都将建立在错误的假设之上。

**核心理论框架：**

1. **业务调研三层框架**（《决胜B端》第3章）：战略层（业务远景、战略目标）→ 战术层（经营策略、组织治理、管控模式）→ 执行层（业务流程、操作细节、合规质检）。产品经理必须从三个层面逐层深入理解业务。

2. **业务调研五环节法**（《决胜B端》3.2节）：明确调研目的 → 选取调研对象 → 确认调研方法 → 执行调研计划 → 总结归纳输出。五个环节缺一不可，调研结束后需输出详细的调研报告。

3. **涉众分析模型**（《决胜B端》3.4节）：识别业务高管、业务经理、一线业务人员、合作伙伴高管、合作伙伴经理、合作伙伴一线人员等不同层级的调研对象，并根据其"影响力"和"兴趣利益"制定差异化的调研策略。

4. **业务问题诊断框架**（《决胜B端》3.3节）：调研的核心产出是"业务现状"和"业务问题"的清晰描述，问题需要分类、分级、排优先级，而非笼统罗列。

## 检查清单

### 1.1 战略层分析

- [ ] **业务背景与现状描述**
  - 检查内容：PRD是否清晰描述了业务背景、行业现状、企业当前面临的核心挑战
  - 优质标准：有具体数据支撑（如"当前月均订单量5万单，但退货率高达15%"），说明了业务处于什么发展阶段（初创期/增长期/成熟期/衰退期）
  - 问题标志：仅有空洞描述（如"随着行业发展，市场竞争加剧"），无具体数据或事实支撑；或照搬行业报告而未结合企业实际情况
  - 参考依据：《决胜B端》PRD模板第1节"项目背景"要求"详细描述背景，包括业务现状，面临问题，解决思路等，需要有数据支持"

- [ ] **战略目标清晰度**
  - 检查内容：是否说明了企业/业务线的战略目标，以及本产品/功能如何服务于该目标
  - 优质标准：明确指出业务负责人的KPI或OKR，并说明产品如何支撑（如"本CRM系统支撑销售团队实现年度GMV增长30%的目标"）
  - 问题标志：仅说"提升效率""优化体验"等模糊目标，无法衡量成功标准
  - 参考依据：《决胜B端》调研三层框架之战略层——了解"业务战略定位、战略目标等信息"

- [ ] **业务负责人诉求与一线用户诉求的区分**
  - 检查内容：是否识别并区分了管理层的管理诉求和一线操作人员的使用诉求
  - 优质标准：分别列出管理层关注点（如管控、数据可视化、合规）和一线关注点（如操作效率、少填字段、移动端支持），并说明两者如何平衡
  - 问题标志：只提管理层需求而忽视一线用户，或只关注操作体验而忽视管理诉求
  - 参考依据：《决胜B端》明确指出B端产品"用户和客户是不同的人"，客户（决策者/买家）和用户（操作者）的诉求往往不同

### 1.2 战术层分析

- [ ] **核心业务流程梳理**
  - 检查内容：是否用简明的方式描述了核心业务主干流程（不超过5-7个关键步骤）
  - 优质标准：用跨部门流程图或泳道图呈现，标注每个环节的负责角色、关键输入输出、系统化程度（线上/线下/半自动）
  - 问题标志：业务流程描述过于笼统（如"用户下单→商家发货→用户收货"缺少中间环节），或过于细碎（一上来就讲字段级别的操作）
  - 参考依据：《决胜B端》2.1节B端产品总体建设流程要求"梳理整个业务主干流程，并确定其中哪些环节需要由该产品实现线上化"

- [ ] **当前业务运作方式**
  - 检查内容：是否描述了业务目前是如何运作的（用什么工具、什么流程、什么痛点）
  - 优质标准：说明当前使用的工具（如Excel、微信、老系统）、当前流程的瓶颈点、数据是如何流转的（是否存在信息孤岛）
  - 问题标志：直接跳到解决方案，没有描述"现在怎么做"和"现在有什么问题"
  - 参考依据：《决胜B端》调研五环节法的输出要求——总结业务现状和问题

- [ ] **组织架构与角色识别**
  - 检查内容：是否梳理了参与业务的所有组织架构和角色关系
  - 优质标准：列出涉及的部门/团队、每个角色的职责范围、角色之间的协作关系和汇报层级，并标注哪些角色是产品的直接用户
  - 问题标志：仅列出"管理员""普通用户"等笼统角色，未深入分析实际业务中的角色分工
  - 参考依据：《决胜B端》3.4节涉众分析——识别不同层级的利益相关者

### 1.3 执行层分析

- [ ] **一线业务操作细节**
  - 检查内容：是否深入到一线业务人员的实际操作场景，了解其日常工作流
  - 优质标准：描述了典型一天的工作流程、高频操作动作、操作环境（PC/移动/多设备切换）、典型的工作量和时间分配
  - 问题标志：对一线操作缺乏描述，或仅基于管理层转述而非直接调研
  - 参考依据：《决胜B端》调研三层框架之执行层——掌握"作业过程、操作细节等信息"

- [ ] **业务痛点的分类与优先级**
  - 检查内容：是否将发现的业务问题进行了分类、分级，并确定解决优先级
  - 优质标准：痛点按影响范围（影响多少人/多少业务）、严重程度（效率损失/资金损失/合规风险）、紧迫度分级，有明确的优先级排序
  - 问题标志：痛点罗列但不分轻重，或只有1-2个笼统痛点（如"系统不好用"）
  - 参考依据：《决胜B端》3.2.1节——调研结束后要"确定各个问题的优先级，以便为后续的方案设计和实施路径提供决策支持"

- [ ] **调研方法与数据来源说明**
  - 检查内容：PRD中的业务描述是否说明了信息来源（访谈谁、观察什么、数据从哪来）
  - 优质标准：明确说明调研方法（访谈/轮岗/问卷）、调研对象（角色/数量）、调研时间范围，关键结论有数据或引述支撑
  - 问题标志：所有业务描述看起来像是产品经理自己的推测，无法追溯信息来源
  - 参考依据：《决胜B端》3.5节业务调研方法——包括定性分析（访谈、轮岗实习）和定量分析（问卷）

### 1.4 政策法规与行业约束

- [ ] **行业法规与合规要求**
  - 检查内容：是否识别了影响产品设计的行业法规、政策要求、合规标准
  - 优质标准：列出相关法规名称和关键条款（如GDPR、等保三级、行业准入资质），说明对产品设计的具体约束
  - 问题标志：涉及金融、医疗、教育等强监管行业但未提及任何合规要求
  - 参考依据：《决胜B端》调研计划表中"政策法规"作为独立调研维度

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| PRD业务背景只有一段话，缺乏深度 | 未做充分的业务调研，或调研结果未写入PRD | 补充调研报告核心结论，至少包含现状、问题、目标三要素 |
| 只有管理层视角，缺少一线用户声音 | 调研对象单一，仅访谈了业务负责人 | 按涉众分析模型补充一线人员、外部合作方的调研 |
| 业务痛点空泛（"系统太慢""体验不好"） | 未深入到执行层了解具体操作场景 | 通过轮岗实习或现场观察获取一手信息 |
| 业务流程描述要么太粗要么太细 | 缺乏"自顶向下"的分层描述方法 | 先用5步描述主干流程，再逐层展开子流程 |
| 多个痛点并列但无优先级 | 缺乏问题分级框架 | 按影响范围x严重程度x紧迫度三维评估排序 |
| 数据和结论缺乏来源标注 | 调研过程不规范，信息管理松散 | 建立调研笔记归档习惯，PRD中标注数据来源 |

# 产品类型适配性检查

> Supporting reference for the `check-prd` skill. 在执行产品类型适配性检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

B端产品种类繁多，不同类型的产品在设计方法、运营模式、技术架构上差异巨大。PRD必须明确产品所属类型，并采用与之匹配的设计策略。错误的产品类型认知会导致设计方向偏差，例如把工具型产品当成业务型产品来设计，会导致过度复杂化。

**核心理论框架：**

1. **B端产品四大分类**（《决胜B端》1.3节）：业务型产品（ERP、CRM等）、工具型产品（电子签名、文档协同等）、交易型产品（电商系统、OMS等）、基础服务产品（权限管理、消息推送等）。四类产品的设计重点和方法论完全不同。

2. **产品部署模式分类**（《决胜B端》1.2.2节）：私有化部署 vs 云部署（SaaS），两种模式对产品化、定制化、安全性要求截然不同。

3. **自研系统 vs 商业化产品**（《决胜B端》第2章）：企业自研自用的系统和商业化对外售卖的产品，在设计思维、客户关系、产品化策略上有根本区别。

4. **B端产品的核心特征**（《决胜B端》1.2.2节）：目标用户是一个群体、用户和客户是不同的人、效能第一体验第二、强调抽象和逻辑、效果受多因素影响。

## 检查清单

### 2.1 产品类型识别

- [ ] **产品类型明确声明**
  - 检查内容：PRD是否明确声明了产品属于哪种类型（业务型/工具型/交易型/基础服务型）
  - 优质标准：在PRD开头或概述部分明确说明产品类型，并解释选择该分类的理由（如"本产品为业务型CRM系统，支撑销售团队的客户管理和商机跟进流程"）
  - 问题标志：PRD通篇没有明确产品定位，或定位模糊（如"一个帮助企业提效的平台"）
  - 参考依据：《决胜B端》1.3节四大产品分类

- [ ] **产品类型与设计方法匹配**
  - 检查内容：PRD中的设计方法和关注重点是否与产品类型匹配
  - 优质标准：
    - 业务型产品：重点关注业务流程、角色协同、数据模型、权限体系
    - 工具型产品：重点关注核心功能效率、操作体验、用完即走的便捷性
    - 交易型产品：重点关注交易流程闭环、支付清结算、商品管理、营销能力
    - 基础服务产品：重点关注接口设计、可扩展性、多业务线复用性、性能
  - 问题标志：设计重点与产品类型不匹配（如工具型产品却花大量篇幅设计复杂审批流程，或基础服务产品却重点设计业务逻辑）
  - 参考依据：《决胜B端》对四类产品特征的描述

- [ ] **单边市场与多边市场的识别**（仅交易型产品）
  - 检查内容：如果是交易型产品，是否明确了商业模型是单边市场还是多边市场（平台模式）
  - 优质标准：明确说明交易关系（企业直销/平台撮合/分销等），并据此设计了对应的角色体系（如多边市场需要设计商户入驻、商户管理功能）
  - 问题标志：交易型产品未说明商业模型，或平台型产品缺少供应商侧的设计
  - 参考依据：《决胜B端》1.3.3节交易型产品的单边市场与多边市场模式

### 2.2 部署模式与技术架构适配

- [ ] **部署模式明确**
  - 检查内容：PRD是否说明了产品的部署模式（私有化/云部署/混合部署）
  - 优质标准：说明部署模式选择的原因（如安全性要求、客户IT环境约束、成本考量），以及对产品设计的影响（如私有化部署需要考虑现场实施和运维）
  - 问题标志：未提及部署模式，或部署模式与客户需求不匹配（如面向银行客户却只考虑SaaS部署）
  - 参考依据：《决胜B端》1.2.2节——私有化部署"安全性强，稳定性好"，云部署"在保证安全性的前提下降低客户的运维成本"

- [ ] **技术架构类型**
  - 检查内容：是否说明了采用B/S还是C/S架构，以及端（PC Web/移动App/小程序/桌面客户端）的选择
  - 优质标准：根据用户使用场景选择合适的端和架构（如一线外勤人员需要移动端，办公室人员以PC Web为主），并说明选择理由
  - 问题标志：未考虑用户实际使用环境就确定了端的选择，或遗漏了关键端（如需要移动办公的业务只设计了PC端）
  - 参考依据：《决胜B端》1.2.2节B/S与C/S架构的区别

### 2.3 自研与商业化定位

- [ ] **产品运营模式定位**
  - 检查内容：PRD是否明确说明产品是企业自研自用还是商业化对外售卖
  - 优质标准：明确说明产品的商业属性，并据此调整设计策略——自研系统以满足内部需求为主、商业化产品需要考虑标准化和通用性
  - 问题标志：定位模糊，在自研和商业化之间摇摆（如"先内部用，以后可能卖给外部客户"而不考虑这对产品架构的影响）
  - 参考依据：《决胜B端》第2章明确区分了自研系统与商业化产品两条设计路径

- [ ] **标准化与个性化策略**（商业化产品必查）
  - 检查内容：如果是商业化产品（特别是SaaS），是否设计了标准化和个性化的平衡策略
  - 优质标准：明确标准功能范围和可配置/可定制的范围，有产品化策略（如通过配置中心、插件机制、开放API来满足个性化需求）
  - 问题标志：SaaS产品完全按照单一客户需求设计，无标准化考量；或过度标准化而忽视客户核心个性化诉求
  - 参考依据：《决胜B端》第8章产品化策略——标准化和个性化之间的平衡

### 2.4 B端产品核心特征合规

- [ ] **用户与客户的区分**
  - 检查内容：PRD是否区分了产品的用户（实际操作者）和客户（购买决策者）
  - 优质标准：分别定义了用户画像和客户画像，说明各自的核心诉求和关注点；对于SaaS产品，还需要考虑"试用者-购买者-使用者-管理者"等不同角色
  - 问题标志：用户和客户混为一谈，或只考虑了其中一方
  - 参考依据：《决胜B端》1.2.2节——"用户和客户是不同的人"是B端产品的核心特征之一

- [ ] **效能优先原则**
  - 检查内容：产品设计是否将业务效能放在首位，而非过度追求视觉体验
  - 优质标准：功能设计以解决业务问题为核心导向，在保证效能的基础上优化体验；不为了视觉美观牺牲操作效率
  - 问题标志：过度追求界面美观而忽视功能完整性；或者反过来，完全不考虑用户体验
  - 参考依据：《决胜B端》1.2.2节——"效能第一，体验第二"，同时《决胜体验设计》补充指出"不是不做体验，而是在效能基础上做好体验"

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| 工具型产品设计了复杂的审批流程 | 未识别产品类型，将工具型误当业务型 | 回归工具型产品核心：聚焦效率，场景单一，用完即走 |
| SaaS产品按单一客户需求定制 | 未理解商业化产品的标准化要求 | 区分标准功能和可配置功能，建立产品化策略 |
| 交易型产品缺少清结算设计 | 对交易型产品的完整能力闭环认知不足 | 补充OMS、支付、清结算、营销等交易核心能力 |
| 只有PC端设计，但用户主要在外勤 | 端选择未基于用户实际使用场景 | 基于用户画像和工作场景重新评估端选择 |
| 产品定位在自研和商业化之间摇摆 | 商业决策不明确，导致设计方向模糊 | 先明确商业定位，再据此确定标准化/定制化策略 |

# 产品定位合理性检查

> Supporting reference for the `check-prd` skill. 在执行产品定位合理性检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

产品定位是连接业务问题和产品方案的关键桥梁。定位不准，方案再精美也是南辕北辙。《决胜B端》强调产品整体方案设计的第一步就是明确产品定位——要解决什么问题、为谁解决、核心价值是什么。《决胜体验设计》进一步提出产品价值主张的概念，强调产品的存在理由和核心用户价值。

**核心理论框架：**

1. **产品整体方案设计要素**（《决胜B端》第4章）：核心业务流程 → 产品定位 → 应用架构 → 功能模块与演进蓝图。产品定位是承上启下的关键环节。

2. **市场分析框架**（《决胜B端》第3章）：产业链分析 → 行业分析 → 市场结构分析（金字塔形/枣核形/倒三角形）→ 目标客群定位 → 种子客户选择。

3. **产品价值主张**（《决胜体验设计》5.1节）：每个产品都应有明确的定位和价值主张，围绕这些原则来构建产品。价值主张回答"为什么用户应该选择你的产品而不是竞品或替代方案"。

4. **用户体验五要素之战略层**（Jesse James Garrett，《决胜体验设计》引用）：软件产品必须先回答自己存在的原因和目的——产品目标和用户需求。

## 检查清单

### 3.1 目标客户定位

- [ ] **目标客户群体定义**
  - 检查内容：PRD是否明确定义了目标客户群体的特征（行业、规模、业态、发展阶段）
  - 优质标准：给出具体的客户画像（如"目标客户为一线城市餐饮连锁企业，门店数量10-100家，年营收500万-5000万"），并说明为什么选择这个群体
  - 问题标志：目标客户描述过于宽泛（如"所有中小企业"）或完全缺失
  - 参考依据：《决胜B端》PRD模板第3节"目标市场与客户分析"——"产品主要针对什么行业，什么市场"

- [ ] **市场结构与SaaS适用性分析**（商业化产品必查）
  - 检查内容：是否分析了目标市场的结构类型，并据此判断产品模式的可行性
  - 优质标准：识别市场结构（金字塔形/枣核形/倒三角形/哑铃形），分析腰部客户的标准化需求是否可收敛；如果是SaaS产品，判断该市场是否适合SaaS模式
  - 问题标志：SaaS产品未分析市场结构，或在倒三角形/哑铃形市场盲目推进SaaS模式
  - 参考依据：《决胜B端》3.1节——枣核形市场"容易做出标准化产品，比较适合SaaS产品"；倒三角形"头部企业个性化强，定制化要求高，很难做出标准化产品"

- [ ] **客户痛点与卖点提炼**
  - 检查内容：是否提炼了目标客户的核心痛点，以及产品的核心卖点
  - 优质标准：痛点具体可衡量（如"目前客户用Excel管理库存，每月盘点误差率超过5%"），卖点能一句话打动客户（如"实时库存追踪，盘点误差率降至0.1%以下"）
  - 问题标志：痛点和卖点空泛，无法感知具体价值（如"提升管理效率"）
  - 参考依据：《决胜B端》PRD模板——"客户痛点：客户遇到的痛点问题有哪些"；"卖点提炼：如果你是销售，如何一句话描述功能亮点，打动客户，让客户付费"

### 3.2 竞品分析

- [ ] **竞品识别的完整性**
  - 检查内容：是否识别了直接竞品、间接竞品和替代方案
  - 优质标准：列出3-5个主要竞品，包括直接竞品（功能类似的产品）、间接竞品（解决同类问题的不同方案）、替代方案（Excel、微信等非专业工具），并分析各自的市场份额
  - 问题标志：未做竞品分析，或只列出竞品名称而无深入分析
  - 参考依据：《决胜B端》PRD模板第3节竞品分析——要求分析竞品商业模式、目标客户、运营策略、市场份额、核心产品价值

- [ ] **竞品差异化分析**
  - 检查内容：是否分析了竞品的优劣势，并明确本产品的差异化定位
  - 优质标准：从功能、流程、交互、价格、服务等维度对比竞品，找到本产品的差异化优势和切入点（如"竞品A功能全面但实施成本高，我们聚焦核心流程快速上线"）
  - 问题标志：竞品分析流于表面（如只比较功能列表），未找到差异化切入点
  - 参考依据：《决胜B端》PRD模板——"竞品相关产品相关分析：比如我们现在要做拼团，可以参考拼多多的产品设计，对比功能、流程、交互视觉等"

### 3.3 产品价值定位

- [ ] **产品定位声明**
  - 检查内容：PRD是否有清晰的产品定位声明——为谁、解决什么问题、提供什么价值
  - 优质标准：用一段话或一个公式明确产品定位（如"本产品是面向中型餐饮连锁企业的供应链管理系统，通过数字化采购和智能库存管理，帮助企业降低食材损耗率30%"）
  - 问题标志：产品定位模糊或缺失，通篇无法一句话概括产品是什么
  - 参考依据：《决胜B端》第4章产品整体方案设计——产品定位是整体方案的核心要素

- [ ] **产品价值主张**
  - 检查内容���是否定义了产品的核心价值主张——用户为什么要用这个产品而不是其他方案
  - 优质标准：价值主张具体、可衡量、有差异性（如"比Excel快10倍的库存管理，比竞品A便宜50%"），并且贯穿PRD始终——所有功能设计都服务于这个价值主张
  - 问题标志：价值主张空洞（如"打造行业领先的数字化平台"），或功能设计偏离价值主张
  - 参考依据：《决胜体验设计》5.1节——产品价值主张是产品设计的灵魂，所有设计决策都应围绕价值主张展开

- [ ] **产品功能边界**
  - 检查内容：是否明确了产品做什么和不做什么的边界
  - 优质标准：列出产品的核心功能范围和明确排除的功能（如"本阶段不做营销模块，专注于交易履约核心流程"），并说明排除的理由
  - 问题标志：产品边界模糊，"什么都要做"或"先做MVP再说"而不明确MVP包含什么
  - 参考依据：《决胜体验设计》引用Jesse五要素之范围层——"软件产品要明确功能规格和功能范围"

### 3.4 项目目标与收益

- [ ] **量化的项目目标**
  - 检查内容：是否设定了符合SMART原则的具体项目目标
  - 优质标准：目标具体（Specific）、可衡量（Measurable）、可达成（Achievable）、相关（Relevant）、有时限（Time-bound），如"上线后3个月内，目标客户系统使用率达到80%"
  - 问题标志：目标模糊（如"提升用户满意度"）或无目标
  - 参考依据：《决胜B端》PRD模板第4节——"具体的量化的项目目标（符合SMART原则），包含验收和成功的标准，以及收益预期，需要量化"

- [ ] **收益预期与度量方式**
  - 检查内容：是否说明了预期的业务收益和如何度量成功
  - 优质标准：定义了核心成功指标（如客户数、续约率、GMV、操作效率提升百分比），并说明数据采集和度量方式
  - 问题标志：无收益预期，或收益预期无法度量
  - 参考依据：《决胜B端》PRD模板——"任何项目都要有业务价值，对于2B产品功能，很多时候难以衡量直接的价值收益，可以考量功能的使用情况，满意度等等"

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| 目标客户是"所有企业" | 缺乏市场细分和聚焦 | 用行业/规模/业态三维度锁定目标客群 |
| 竞品分析只是功能列表对比 | 未从商业模式和用户价值层面分析 | 补充竞品的商业模式、目标客群、核心价值对比 |
| 产品"什么都要做" | 产品边界不清，缺少聚焦 | 先明确核心价值主张，围��价值主张划定功能边界 |
| 项目目标是"上线" | 把交付当目标，缺少业务价值导向 | 定义业务层面的成功标准，而非技术层面的交付标准 |
| SaaS产品在强监管行业盲目推进 | 未做市场结构分析 | 分析市场结构类型，评估SaaS模式可行性 |

# 场景分析与用户旅程检查

> Supporting reference for the `check-prd` skill. 在执行场景分析与用户旅程检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

场景分析和用户旅程是连接"业务需求"与"功能设计"的核心纽带。《决胜B端》第6章提出了"基于场景的需求分析和建模"方法论，强调所有功能设计都应从场景出发。《决胜体验设计》进一步深化了用户旅程地图的方法，将体验设计融入场景分析中。

**核心理论框架：**

1. **需求挖掘十三要素五步法**（《决胜B端》6.2节）：通过五个核心步骤，覆盖十三个关键要素，帮助全面梳理需求、挖掘本质问题：
   - **步骤一：分析相关角色**——要素1提出人、要素2使用人、要素3受影响人
   - **步骤二：了解基本场景**——要素4基本场景（人物+时间+地点+起因+经过+结果）、要素5发生频率
   - **步骤三：挖掘真实动机**——要素6核心诉求/痛点（用5Why法深挖）、要素7强烈程度
   - **步骤四：发散更多场景**——要素8实际价值、要素9横向替代场景（MECE穷举）、要素10纵向互补场景（上下游动线）
   - **步骤五：设计产品方案**——要素11已有方案、要素12功能需求、要素13非功能需求

2. **场景六要素**（《决胜B端》PRD模板第2节）：人物、时间、地点、起因、经过、结果——完整描述一个业务场景的六个必备要素。

3. **客户旅程地图**（《决胜体验设计》4.2节）：从客户视角出发，梳理完整的业务旅程，识别每个触点的体验问题和改进机会。旅程地图是客户体验设计CX的经典研究工具。

4. **用户体验四层模型**（《决胜体验设计》1.1节）：界面设计UI → 用户体验设计UX → 客户体验设计CX → 服务设计SD。场景分析应覆盖从人机交互到服务全流程的完整体验。

5. **B端用户分类模型**（《决胜体验设计》2.1节）：按角色分为决策者/管理者/执行者/协作者，不同角色的使用场景和体验需求完全不同。

## 检查清单

### 4.1 用户角色定义

- [ ] **用户角色识别的完整性**
  - 检查内容：是否识别了所有与产品交互的用户角色，包括直接用户和间接用户
  - 优质标准：列出至少3种以上用户角色（如管理员、普通操作员、审批者、查看者、外部合作方），每个角色有明确的定义和职责描述
  - 问题标志：只有"管理员"和"普通用户"两种笼统角色，或只考虑了内部用户而忽视外部角色
  - 参考依据：《决胜B端》B端产品特征——"目标用户是一个群体"，需要支撑"分工协作"

- [ ] **用户画像的具体性**
  - 检查内容：是否为核心用户角色建立了具体的用户画像（Persona）
  - 优质标准：画像包含角色名称、岗位职责、技术熟练度、使用频率、核心任务、痛点、期望；最好有拟人化的角色描述（如"张经理，35岁，区域销售主管，管理15人团队"）
  - 问题标志：无用户画像，或画像过于抽象（如"用户是企业中的管理人员"）
  - 参考依据：《决胜体验设计》2.2节——用户画像是体验设计的基础输入

- [ ] **决策者/管理者/执行者/协作者的区分**
  - 检查内容：是否区分了不同层级用户的角色类型和使用场景
  - 优质标准：
    - 决策者（CxO/VP）：关注数据看板、经营指标，低频使用
    - 管理者（总监/经理）：关注团队管理、流程监控、审批，中频使用
    - 执行者（一线员工）：关注日常操作效率、任务完成，高频使用
    - 协作者（跨部门/外部）：关注信息同步、协作效率
  - 问题标志：所有用户被视为同一类型，功能设计不区分使用层级
  - 参考依据：《决胜体验设计》2.1节B端用户分类模型

### 4.2 场景描述质量

- [ ] **核心场景的完整描述**
  - 检查内容：是否用场景六要素完整描述了核心业务场景
  - 优质标准：每个核心场景都包含人物（谁在操作）、时间（什么时候操作）、地点（在什么环境操作）、起因（为什么要做这件事）、经过（具体操作步骤）、结果（完成后产生什么结果）
  - 问题标志：场景描述缺少关键要素（如只说"用户提交订单"而不说谁、在什么情况下、提交后发生什么）
  - 参考依据：《决胜B端》PRD模板第2节——"场景（人物、时间、地点、起因、经过、结果）"

- [ ] **场景覆盖的充分性**
  - 检查内容：是否覆盖了正常场景、异常场景和边界场景
  - 优质标准：
    - 正常场景：至少覆盖主流程的所有关键环节
    - 异常场景：至少列出3种以上异常情况（如网络中断、数据冲突、权限不足、操作超时）
    - 边界场景：识别了极端条件（如大批量数据操作、并发操作、跨时区问题）
  - 问题标志：只描述了理想路径（Happy Path），完全没有异常和边界场景考虑
  - 参考依据：《决胜B端》PRD模板10.3节——"异常情况的处理方案：例如断网、断电、误操作、数据丢失等"

- [ ] **场景的发生频率与优先级**
  - 检查内容：是否标注了各个场景的发生频率和业务重要性
  - 优质标准：标注了每个场景的发生频率（高频/中频/低频）和业务影响（高/中/低），据此确定功能优先级
  - 问题标志：所有场景平等对待，没有优先级区分
  - 参考依据：《决胜B端》PRD模板第2节——"发生频率"是需求基本情况的必填项

### 4.3 用户旅程设计

- [ ] **端到端用户旅程**
  - 检查内容：是否绘制了核心角色的端到端用户旅程（从触发业务需求到完成任务的完整路径）
  - 优质标准：旅程图包含阶段划分、每阶段的用户行为、接触渠道（线上/线下/电话）、情绪曲线、痛点标注和改进机会点
  - 问题标志：只有功能列表而没有用户视角的完整旅程描述
  - 参考依据：《决胜体验设计》4.2节客户旅程地图——梳理完整的业务旅程，识别每个触点的体验问题

- [ ] **线上线下场景衔接**
  - 检查内容：B端业务往往涉及线上线下结合，PRD是否考虑了两者的衔接
  - 优质标准：标注了哪些环节在线上完成、哪些在线下完成、线上线下如何衔接（如"销售在移动端录入客户信息后，由后台运营人员在PC端审核"）
  - 问题标志：只考虑了系统内的操作流程，忽视了线下环节和人工操作
  - 参考依据：《决胜体验设计》体验四层模型中CX层——关注用户接触服务的全部场景

- [ ] **多角色协作旅程**
  - 检查内容：是否描述了不同角色在同一业务流程中的协作方式
  - 优质标准：用泳道图或协作流程图展示多角色如何在同一流程中配合，明确角色之间的信息传递和任务流转
  - 问题标志：每个角色的操作被孤立描述，看不出角色之间如何协作
  - 参考依据：《决胜B端》6.3节跨部门流程图——使用频率最高的UML图，专门用于描述多角色协作

### 4.4 场景与功能的映射

- [ ] **场景到功能的追溯性**
  - 检查内容：每个功能是否都能追溯到具体的业务场景和用户需求
  - 优质标准：功能列表中每个功能点都标注了对应的场景编号或需求编号，能回答"为什么要做这个功能"
  - 问题标志：存在"无源功能"——无法追溯到任何业务场景的功能设计
  - 参考依据：《决胜B端》产品经理本质——"必须能够定义问题、解决问题、为价值交付负责"

- [ ] **需求提出人和功能使用人的区分**
  - 检查内容：是否区分了需求提出人（谁要求做这个功能）和功能使用人（谁实际使用这个功能）
  - 优质标准：每个功能都标注了需求提出人和实际使用人，当两者不同时有特别说明（如"需求由运营总监提出，实际由一线运营人员使用"）
  - 问题标志：不区分提出人和使用人，导致功能设计偏向提出人（管理者）而非使用人（一线操作者）
  - 参考依据：《决胜B端》PRD模板第2节明确区分了"需求提出人"和"功能使用人"

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| 场景只有Happy Path | 产品经理缺乏对异常情况的思考 | 对每个核心场景做"如果...会怎样"的异常推演 |
| 用户角色只有管理员和用户 | 未做深入的涉众分析 | 按决策者/管理者/执行者/协作者四层模型重新分析 |
| 功能列表无法追溯到场景 | 需求分析不够系统，功能是"拍脑袋"想出的 | 建立场景→需求→功能的追溯矩阵 |
| 线下环节完全缺失 | 只关注系统设计，忽视了业务全流程 | 画端到端用户旅程图，标注线上线下环节 |
| 多角色协作关系不清楚 | 未使用泳道图等协作可视化工具 | 绘制跨部门/跨角色泳道图 |
| 场景描述过于简略 | 未使用场景六要素框架 | 用"人物+时间+地点+起因+经过+结果"完整描述每个场景 |

# 文档结构完整性检查

> Supporting reference for the `check-prd` skill. 在执行文档结构完整性检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

PRD文档的结构完整性直接影响评审效率和开发交付质量。一份结构清晰、章节完整的PRD能降低沟通成本、减少需求遗漏。《决胜B端PRD模板v2.0》提供了标准化的PRD文档结构，共包含14个章节。

**核心理论框架：**

1. **PRD标准文档结构**（《决胜B端PRD模板v2.0》）：项目背景 → 需求基本情况 → 商业分析 → 项目收益目标 → 项目方案概述 → 项目范围 → 项目风险 → 术语和缩略语 → 参考文献 → 功能需求（含产品框架概述、需求详解、异常处理） → 数据埋点 → 角色和权限 → 运营计划 → 待决事项。

2. **文档版本管理**（《决胜B端PRD模板v2.0》）：PRD交付前必须标记版本号，每次修改严格记录变更时间、原因、修改人、审核人。

3. **功能描述五种规则类型**（《决胜B端PRD模板v2.0》10.2节）：事实、约束、触发条件、推论、计算——功能规则描述必须归类到这五种类型之一，确保无遗漏。

4. **自顶向下的设计思路**（《决胜B端》第2章）：B端产品总体建设遵循"从抽象到具体"的原则，PRD文档结构应体现这一思路。

## 检查清单

### 5.1 文档元信息

- [ ] **文档头部信息完整性**
  - 检查内容：PRD是否包含审核人、重要性、紧迫性、需求方、编写人、提交日期等元信息
  - 优质标准：所有字段已填写，审核人和需求方明确到具体人名，日期格式统一
  - 问题标志：元信息缺失或填写不完整（如审核人写"待定"）
  - 参考依据：《决胜B端PRD模板v2.0》文档头部格式

- [ ] **版本变更记录**
  - 检查内容：是否维护了完整的PRD版本变更记录
  - 优质标准：每次修改都记录了变更时间、变更内容、变更提出部门与理由、修改人、审核人、版本号；版本号递增合理
  - 问题标志：无版本记录，或修改后未更新版本号，或变更记录不完整
  - 参考依据：《决胜B端PRD模板v2.0》明确指出"文档的版本管理是非常重要的工作，任何的文档修改都需要在文档中准确的记录"

### 5.2 前置章节完整性（第1-9节）

- [ ] **项目背景（第1节）**
  - 检查内容：是否有详细的项目背景描述
  - 优质标准：包含业务现状、面临问题、解决思路，有数据支持
  - 问题标志：背景空泛或缺失
  - 参考依据：PRD模板第1节——"不论是多简单的需求，都会有需求背景"

- [ ] **需求基本情况（第2节）**
  - 检查内容：是否包含需求提出人、功能使用人、受影响人、场景、发生频率、核心痛点、需求价值
  - 优质标准：七个字段全部填写，场景用六要素描述（人物、时间、地点、起因、经过、结果）
  - 问题标志：多个字段缺失，或场景描述过于简略
  - 参考依据：PRD模板第2节需求基本情况

- [ ] **商业分析（第3节）**
  - 检查内容：是否包含目标市场与客户分析、竞品分析两大板块
  - 优质标准：目标市场分析包含目标市场、市场规模、市场特征、发展趋势、客户画像、客户痛点、卖点提炼；竞品分析包含商业模式、目标客户、运营推广策略、市场份额、产品分析、核心价值
  - 问题标志：商业分析章节缺失或仅有空壳
  - 参考依据：PRD模板第3节商业分析表格

- [ ] **项目收益目标（第4节）**
  - 检查内容：是否有量化的、符合SMART原则的项目目标
  - 优质标准：目标具体可衡量，包含验收标准和成功标准，有收益预期
  - 问题标志：目标模糊或缺失
  - 参考依据：PRD模板第4节

- [ ] **项目方案概述（第5节）**
  - 检查内容：是否用简要语言概述了核心功能或项目方案
  - 优质标准：列举了所有核心功能feature，或概述了产品方案、运营方案、技术方案
  - 问题标志：无概述直接进入细节
  - 参考依据：PRD模板第5节

- [ ] **项目范围（第6节）**
  - 检查内容：是否明确了项目涉及的系统、产品、接口和影响范围
  - 优质标准：列出所有相关系统和责任方，标注影响范围，确保无遗漏
  - 问题标志：范围不明确，后续开发时才发现遗漏的关联方
  - 参考依据：PRD模板第6节——"梳理项目范围，提前确认项目关联方、影响方"

- [ ] **项目风险（第7节）**
  - 检查内容：是否识别并记录了假设、约束和各类风险
  - 优质标准：列出产品风险、运营风险、技术风险，每个风险都有应对方案
  - 问题标志：无风险分析，或只列风险不提应对方案
  - 参考依据：PRD模板第7节

- [ ] **术语和引用文档（第8-9节）**
  - 检查内容：是否解释了文档中的专业术语和缩略语，是否列出了参考文档
  - 优质标准：所有首次出现的术语有定义；引用的外部文档有明确链接
  - 问题标志：文档中充满缩略语但无解释，或引用了但找不到参考文档
  - 参考依据：PRD模板第8-9节——"尽量不要自己定义缩略语"

### 5.3 功能需求章节（第10节）

- [ ] **产品框架概述（10.1节）**
  - 检查内容：是否包含系统框架图、数据模型图、业务流程图、状态机图中的至少2-3种
  - 优质标准：四种图都有提供，框架图层次清晰，数据模型用ER图表达，流程图用跨部门泳道图表达
  - 问题标志：无任何框架概述图，直接进入页面级描述
  - 参考依据：PRD模板10.1节——"系统框架图、数据模型图、业务流程图、状态机图"

- [ ] **功能清单表格（10.1.2节）**
  - 检查内容：是否提供了子系统、页面、多端覆盖情况的功能清单
  - 优质标准：功能清单以表格形式呈现，包含子系统名称、页面名称、各端支持情况（PC/H5/App等），一目了然
  - 问题标志：无功能清单，或功能列举散落在文档各处
  - 参考依据：PRD模板10.1.2节功能清单表格格式

- [ ] **需求详解的规范性（10.2节）**
  - 检查内容：功能需求描述是否采用结构化方式（查询条件、列表字段、规则描述）
  - 优质标准：
    - 列表页有查询条件表格（字段名称、默认值、字段类型、备注）
    - 列表页有列表字段表格（字段名称、默认值、字段类型、开放修改、必输项、备注）
    - 规则描述按五种类型归类：事实、约束、触发条件、推论、计算
    - 每个功能模块有业务流程图和页面交互说明
  - 问题标志：功能描述为大段自然语言叙述，无结构化表格和分类
  - 参考依据：PRD模板10.2节的结构化描述格式和五种规则类型

- [ ] **异常处理方案（10.3节）**
  - 检查内容：是否有专门的异常情况处理方案章节
  - 优质标准：覆盖断网、断电、误操作、数据丢失、并发冲突等异常场景，每种异常有处理方案
  - 问题标志：无异常处理章节，或仅提"参见后续补充"
  - 参考依据：PRD模板10.3节

### 5.4 后置章节完整性（第11-14节）

- [ ] **数据埋点（第11节）**
  - 检查内容：是否按公司统一标准描述了需要监控的按钮、页面、事件
  - 优质标准：埋点设计覆盖核心操作路径，标注了事件名称、触发条件、采集参数
  - 问题标志：无埋点设计，或只有"后续补充"的占位符
  - 参考依据：PRD模板第11节

- [ ] **角色和权限（第12节）**
  - 检查内容：是否以权限矩阵的形式描述了各角色在各页面/功能上的权限
  - 优质标准：用表格形式列出一级导航、页面、页面元素与各角色的权限对应关系，精确到按钮级别
  - 问题标志：权限描述模糊（如"管理员有所有权限"），未精确到页面元素级别
  - 参考依据：PRD模板第12节角色权限矩阵格式

- [ ] **运营计划（第13节）**
  - 检查内容：是否说明了需要建立的运营流程、推广计划、部署计划
  - 优质标准：列出新建运营流程、客服处理流程、推广计划、部署方案
  - 问题标志：完全没有运营视角的考量
  - 参考依据：PRD模板第13节

- [ ] **待决事项（第14节）**
  - 检查内容：是否明确列出了文档中结论待定的事项
  - 优质标准：待决事项有编号、描述、负责人、预计决策时间，且数量不应过多（过多说明PRD成熟度不够）
  - 问题标志：明显有未决事项但未记录，或待决事项数量超过10项
  - 参考依据：PRD模板第14节

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| PRD只有功能描述，缺少背景和分析 | 把PRD当功能说明书写 | 按14节标准结构补充前9节 |
| 功能描述是大段文字 | 不熟悉结构化描述方法 | 采用表格+规则分类的标准格式 |
| 无版本记录 | 缺乏文档管理意识 | 建立版本管理规范，每次修改必记录 |
| 权限描述模糊 | 未做角色权限矩阵 | 按模板格式建立角色-页面-元素权限矩阵 |
| 待决事项过多 | PRD尚未成熟就匆忙评审 | 先解决核心待决事项再提交评审 |
| 缺少框架概述图 | 直接跳入细节设计 | 先完成系统框架图、ER图、流程图 |

# 架构设计质量检查

> Supporting reference for the `check-prd` skill. 在执行架构设计质量检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

架构设计是产品的骨架，决定了系统的扩展性、稳定性和与企业整体IT体系的融合程度。本维度分两层检查：

- **通用层（6.1-6.4）**：适用于所有 PRD，检查单系统维度的架构质量
- **企业架构层（6.5-6.9）**：仅适用于 **0-1 新系统设计或系统级规划 PRD**。如果 PRD 是迭代需求或模块级需求，6.5-6.9 标注"不适用"并跳过

**核心理论框架：**

1. **产品整体方案设计要素**（《决胜B端》第4章）：核心业务流程 → 产品定位 → 应用架构 → 功能模块与演进蓝图。应用架构是连接产品定位和功能细节的关键层。

2. **应用架构设计**（《决胜B端》4.2-4.3节）：包括系统间的关系（与公司现有产品体系如何融合）、系统内的功能模块划分、模块之间的依赖关系。

3. **企业应用版图规划**（《决胜B端》第16章）：将公司所有系统按业务能力域组织成一张地图，识别能力重叠、系统空白，作为新系统定位的依据。

4. **数据架构设计**（《决胜B端》产品规划部分 + 第16章）：明确企业数据域的划分，防止同一数据在多个系统中各自维护、互相矛盾。

5. **基础服务抽象**（《决胜B端》1.3.4节）：识别各系统共性需求（权限管理、消息推送、MDM、SSO等），抽象为基础服务，避免重复建设。

6. **功能架构与信息架构**（《决胜体验设计》第5章）：功能架构关注产品能力的结构化组织，信息架构关注内容和数据的组织方式。

## 【通用层】适用于所有 PRD

### 6.1 功能架构设计

- [ ] **功能架构图的层次性**
  - 检查内容：是否提供了产品的功能架构图，并体现层次结构
  - 优质标准：功能架构图从上到下分为 3-4 层（如：导航大类 → 功能模块 → 功能页面 → 核心功能点），层次清晰，一目了然
  - 问题标志：无功能架构图，或架构图只有一层（功能点平铺），看不出层次关系
  - 参考依据：《决胜体验设计》第5章功能架构设计

- [ ] **功能完整性与闭环**
  - 检查内容：功能设计是否构成完整的业务闭环，无"断头路"
  - 优质标准：从业务发起到业务结束，所有环节都有对应的功能支撑；数据在系统内流转无断点；每个操作都有明确的后续步骤
  - 问题标志：存在功能"断头路"——某个操作执行后，下一步无法继续（如创建了订单但没有发货功能，或审批通过后无后续处理流程）
  - 参考依据：《决胜B端》产品设计——"先做加法再做减法"，确保业务闭环后再做裁剪

- [ ] **功能菜单编排的合理性**
  - 检查内容：产品的功能菜单组织是否合理，用户是否能快速找到所需功能
  - 优质标准：菜单组织遵循业务逻辑或用户心智模型（如按业务阶段、按角色职能、按操作频率组织），层级不超过 3 层，高频功能在前
  - 问题标志：菜单按开发模块而非用户视角组织，层级过深（超过 4 层），高频功能隐藏在深层菜单中
  - 参考依据：《决胜体验设计》引用 Jesse 五要素之结构层——"让用户理解软件，找到所需"

### 6.2 系统边界与外部关系

- [ ] **系统架构图的存在与清晰度**
  - 检查内容：PRD 是否提供了系统架构图，展示产品在企业 IT 体系中的位置
  - 优质标准：架构图展示了本系统与周边系统的关系（上下游系统、依赖的基础服务、外部对接系统），标注了系统间的数据流向和交互方式（API/消息队列/文件等）
  - 问题标志：无架构图，或架构图只画了本系统内部结构而未考虑与外部系统的关系
  - 参考依据：《决胜B端》第4章——"规划产品的功能范围、定位，以及和公司现有产品体系如何融合"

- [ ] **基础服务的识别与复用**
  - 检查内容：是否识别了可以复用的基础服务（权限、消息、组织架构、SSO 等）
  - 优质标准：明确了哪些能力由公司现有基础服务提供，哪些需要自建，避免重复造轮子
  - 问题标志：每个功能模块都自建权限管理、消息推送等通用能力
  - 参考依据：《决胜B端》1.3.4节基础服务产品——MDM、Auth、Org、Push、SSO 等

- [ ] **系统集成与接口设计**
  - 检查内容：如果需要与外部系统集成，是否描述了接口设计
  - 优质标准：列出所有外部接口，每个接口说明了调用方向（主动/被动）、数据格式、频率、异常处理方式
  - 问题标志：提到了需要与外部系统集成但未设计接口，或接口描述不完整
  - 参考依据：《决胜B端》PRD 模板第6节项目范围——涉及哪些系统和接口

### 6.3 模块设计质量

- [ ] **功能模块划分的合理性**
  - 检查内容：系统内部的功能模块划分是否合理，模块之间的边界是否清晰
  - 优质标准：模块划分遵循高内聚低耦合原则；每个模块有明确的职责定义；模块之间的依赖关系清晰且无循环依赖
  - 问题标志：模块划分过细（导致模块间频繁交互）或过粗（一个模块承担过多职责）；模块边界模糊，同一功能在多个模块中重复
  - 参考依据：《决胜B端》应用架构设计原则

- [ ] **多端架构设计**
  - 检查内容：如果产品需要支持多端（PC Web、移动 App、小程序），是否有统一的多端架构设计
  - 优质标准：明确各端的功能差异和互补关系（如 PC 端做复杂操作，移动端做轻量级操作和审批），后端服务统一，前端按端特性差异化设计
  - 问题标志：多端设计各自为政，未考虑数据一致性和功能互补性
  - 参考依据：《决胜B端》PRD 模板 10.1.2 节功能清单按端标注

### 6.4 技术约束与演进

- [ ] **性能要求与约束**
  - 检查内容：PRD 是否描述了关键的性能要求（响应时间、并发量、数据量级）
  - 优质标准：标注了核心操作的响应时间要求，预估了用户并发规模和数据增长趋势
  - 问题标志：无任何性能要求说明，或要求不切实际
  - 参考依据：《决胜B端》第9章B端产品经理应具备的技术常识

- [ ] **架构的可扩展性与配置化**
  - 检查内容：架构设计是否为未来扩展预留了空间，业务规则是否设计为可配置
  - 优质标准：识别了可能的扩展方向，架构设计不会因扩展而需要大规模重构；将可变的业务规则（审批流、状态流转、字段校验规则）设计为可配置而非硬编码
  - 问题标志：架构紧密耦合，任何新增功能都需要修改核心模块；业务规则硬编码
  - 参考依据：《决胜B端》第8章产品化——通过配置中心、插件机制满足个性化需求

## 【企业架构层】仅适用于 0-1 新系统设计或系统级规划 PRD

> 如果 PRD 是迭代需求或模块级需求，以下 6.5-6.9 标注"不适用"并跳过。

### 6.5 企业应用版图定位

- [ ] **系统在企业 IT 版图中的位置**
  - 检查内容：PRD 是否说明了本系统在公司整体应用版图中的位置
  - 优质标准：说明了本系统所属的业务域（如"营销域"），与哪些现有系统存在功能重叠的边界，以及为什么选择新建而非扩展现有系统
  - 问题标志：新建系统与某个已有系统功能高度重叠但未说明区别；没有交代本系统与已有系统的关系
  - 参考依据：《决胜B端》第16章——"先理清家底，再决定建什么"

- [ ] **业务能力域划分**
  - 检查内容：是否明确了本系统负责哪些业务能力，不负责哪些
  - 优质标准：用业务能力（而非功能模块）描述系统边界，业务能力划分清晰，无模糊地带
  - 问题标志：系统边界描述含混，或一个系统试图覆盖多个能力域
  - 参考依据：《决胜B端》第16章业务能力地图方法论

- [ ] **make-or-buy 决策依据**
  - 检查内容：是否说明了为什么选择自研而非采购或复用
  - 优质标准：评估了市场上是否有成熟解决方案，说明了自研、采购 SaaS、定制化开发三种路径的选择逻辑
  - 问题标志：完全没有提及是否考虑过采购/复用；自研决策缺乏理由
  - 参考依据：《决胜B端》第16章企业IT建设的核心决策

### 6.6 数据架构设计

- [ ] **数据域划分与数据归属**
  - 检查内容：是否明确了本系统的数据边界——哪些数据由本系统负责管理（System of Record），哪些数据来自其他系统
  - 优质标准：列出了本系统的核心数据实体并标注数据主权；对于从外部系统获取的数据，说明了同步策略（实时读取 / 定期同步 / 本地副本）；避免同一数据在多个系统中各自维护
  - 问题标志：数据归属不清楚；同一业务实体（如"客户"）在多个系统中都有独立的完整数据集，存在一致性风险
  - 参考依据：《决胜B端》数据架构设计——"每份数据都应该有唯一的 System of Record"

- [ ] **主数据管理（MDM）策略**
  - 检查内容：如果涉及主数据（客户、产品、员工、组织等），是否说明了主数据的管理策略
  - 优质标准：明确了主数据的权威来源系统；说明了其他系统如何接入主数据（API 查询 / 数据同步）；有主数据变更时的通知机制
  - 问题标志：多个系统各自维护客户或产品数据，没有单一的主数据源
  - 参考依据：《决胜B端》1.3.4 节 MDM（主数据管理）作为基础服务的必要性

- [ ] **跨系统数据一致性**
  - 检查内容：跨系统的数据一致性问题是否被识别并设计了应对策略
  - 优质标准：识别了哪些数据需要在多个系统间保持一致，设计了最终一致性或强一致性方案；有数据对账和异常修复机制
  - 问题标志：存在明显的跨系统数据一致性需求但 PRD 未提及，或简单地说"通过接口同步"而无详细设计
  - 参考依据：《决胜B端》第16章数据架构治理

### 6.7 模块抽象与平台化决策

- [ ] **共享能力识别与抽象决策**
  - 检查内容：是否识别了本系统中哪些能力适合抽象为共享服务
  - 优质标准：识别了可被多个业务系统复用的通用能力（如规则引擎、审批流引擎、报表引擎）；对每个候选共享能力评估了抽象必要性（复用≥2次才值得抽象）；说明了抽象 vs. 内聚的决策依据
  - 问题标志：每个业务系统都各自实现审批流、消息推送等通用能力；或将不够通用的能力强行抽象为平台
  - 参考依据：《决胜B端》第16章模块抽象决策原则

- [ ] **平台化陷阱规避**
  - 检查内容：如果涉及平台化设计，是否规避了常见陷阱
  - 优质标准：平台的使用方明确，接入成本合理；有明确的平台治理机制（接入申请、版本管理、废弃策略）
  - 问题标志：平台设计过于抽象导致没有业务方愿意接入；或接入成本极高高于自建成本
  - 参考依据：《决胜B端》第16章平台化要避免的三大陷阱

### 6.8 集成治理

- [ ] **集成模式选择合理性**
  - 检查内容：与外部系统的集成采用了合适的集成模式
  - 优质标准：根据场景选择了合适的集成模式——实时 API（低延迟场景）、异步消息/事件驱动（松耦合最终一致）、批量 ETL（数据量大时效低）；不推荐直接跨库读取
  - 问题标志：所有集成都用实时 API 调用（高耦合、级联故障风险）；或业务系统直接读取另一个系统的数据库
  - 参考依据：《决胜B端》第16章集成架构设计

- [ ] **API 契约设计**
  - 检查内容：对外暴露的 API 是否有明确的契约设计
  - 优质标准：定义了 API 版本管理策略；说明了向后兼容原则和废弃策略；明确了哪些是稳定 API（对外承诺不变）哪些是内部 API
  - 问题标志：API 无版本管理，随时可能破坏性变更；对外部系统的依赖没有契约保障
  - 参考依据：《决胜B端》第16章 API 治理

### 6.9 技术治理一致性

- [ ] **技术选型与公司标准一致性**
  - 检查内容：本系统的技术选型是否与公司技术栈保持一致
  - 优质标准：在公司已有技术标准范围内做选择；如有偏差，有明确理由说明
  - 问题标志：引入了公司现有团队不熟悉的全新技术栈，增加学习成本和维护风险
  - 参考依据：《决胜B端》第16章技术治理——降低技术异质性

- [ ] **数据治理策略**
  - 检查内容：是否考虑了数据质量和数据治理问题
  - 优质标准：定义了核心数据实体的质量标准；有数据质量检测和修复机制；重要数据有数据血缘追踪；符合公司数据目录和元数据管理规范
  - 问题标志：数据质量无人负责；数据来源不清晰；数据出了问题无法追溯
  - 参考依据：《决胜B端》第16章数据治理框架

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| 无系统架构图 | 直接进入功能设计，缺少宏观视角 | 先画应用架构图，明确系统边界和外部关系 |
| 功能有"断头路" | 未做端到端流程验证 | 沿核心业务流程走查，确保每个环节都有功能支撑 |
| 菜单层级过深用户找不到 | 按技术模块而非用户心智组织菜单 | 按业务流程或用户角色重新组织功能菜单 |
| 每个系统都自建通用能力 | 未识别和复用基础服务 | 梳理公司现有基础服务，能复用的不重复建设 |
| 新系统和老系统功能重叠 | 未做应用版图盘点 | 先梳理公司应用版图，明确新系统的差异化定位 |
| 同一数据在多个系统各自维护 | 未设计数据归属和 MDM 策略 | 明确每类数据的 System of Record，其他系统通过 API/订阅获取 |
| 系统间耦合严重，牵一发动全身 | 集成模式选择不当 | 引入事件驱动集成，减少直接依赖 |
| 平台建好没人用 | 平台抽象过度，接入成本高于自建 | 平台设计要以使用方为中心，降低接入成本 |
| 架构扩展困难 | 初始设计过度耦合，未预留扩展点 | 识别未来可能的扩展方向，在核心模块预留扩展接口 |

# 数据建模质量检查

> Supporting reference for the `check-prd` skill. 在执行数据建模质量检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

数据建模是B端产品设计中最核心、最具技术含量的工作之一。《决胜B端》第5章通过完整的对话案例（老马与小李的ER建模讨论），深入阐述了数据建模的方法论和思维过程。ER建模的质量直接决定了产品的灵活性和可扩展性——一旦数据库表结构定型，后续修改成本将指数级增加。

**核心理论框架：**

1. **ER建模方法论**（《决胜B端》5.1节）：通过ER图（Entity-Relationship图）描述业务实体和实体间的关系。ER建模是"帮你梳理业务的过程"，必须思考得"非常缜密、全面"。

2. **实体抽象原则**（《决胜B端》5.1节案例）：对业务对象进行合理抽象，避免冗余实体。案例中将"家长"和"孩子"抽象为"人"的维度，通过属性标记区分——体现了"先做加法再做减法"的建模思想。

3. **对应关系类型**（《决胜B端》6.3.1节）：一对一（1:1）、一对多（1:*）、多对多（*:*），每种关系都有特定的业务含义和实现方式。UML格式下用数字标注关系（如"0..*"表示0到多个）。

4. **数据模型的前瞻性**（《决胜B端》5.1节）：ER建模必须考虑未来可能的业务场景。"如果一上来就按照最简单的方案设计，未来想切换到理想模型，研发投入会非常巨大"。

5. **主数据管理MDM**（《决胜B端》1.3.4节）：企业在很多业务领域和主体中，应该有且只有一套相关的资料存储系统，保证数据管理的一致性和唯一性。

## 检查清单

### 7.1 ER图设计质量

- [ ] **ER图的存在与完整性**
  - 检查内容：PRD是否提供了核心业务实体的ER图
  - 优质标准：ER图覆盖了所有核心业务实体（如用户、组织、订单、商品、客户等），每个实体有关键属性说明，实体间关系用数字标注（1:1、1:*、*:*）
  - 问题标志：无ER图，或ER图只画了2-3个实体而业务涉及十几个核心对象
  - 参考依据：《决胜B端》PRD模板10.1.1节要求提供数据模型图

- [ ] **实体识别的完整性**
  - 检查内容：是否识别了所有核心业务实体，无重要实体遗漏
  - 优质标准：通过业务场景反推需要的实体——每个业务场景中的"主语"和"宾语"都应该有对应的实体；实体之间的关系能支撑所有已描述的业务场景
  - 问题标志：某些业务场景无法在数据模型上找到支撑（如PRD描述了"一个客户下有多个门店"，但ER图中客户和门店没有关联关系）
  - 参考依据：《决胜B端》5.1节——ER建模过程就是帮你梳理业务的过程

- [ ] **实体抽象的合理性**
  - 检查内容：实体的抽象层级是否合理，既不过度抽象也不过度具体
  - 优质标准：相似的业务对象有适当抽象（如将"家长"和"学生"抽象为"人"），但抽象不影响理解和实现；每个实体都有明确的业务含义
  - 问题标志：过度抽象（如将所有东西都抽象为"对象"，失去业务语义）；或不抽象（如家长、学生、教师分别建模，产生大量冗余实体和重复字段）
  - 参考依据：《决胜B端》5.1节老马与小李的对话——最终将家长和孩子抽象到"人"的维度

### 7.2 实体关系设计

- [ ] **关系类型的准确性**
  - 检查内容：实体间的对应关系类型是否准确反映了业务规则
  - 优质标准：
    - 一对一（1:1）：确实是业务约束（如一个账号对应一个员工）
    - 一对多（1:*）：方向和约束正确（如1个部门包含多个员工，而非反向）
    - 多对多（*:*）：有中间表来承载关系，中间表是否需要额外属性
  - 问题标志：关系类型标注不准确或缺失（如一对多画成了一对一），或多对多关系缺少中间实体
  - 参考依据：《决胜B端》6.3.1节ER图详解——实线两端标数字描述对应关系

- [ ] **关系的下限约束**
  - 检查内容：是否明确了关系的下限（0还是1）
  - 优质标准：区分了"0到多"（0..*，即可以没有）和"1到多"（1..*，即至少有一个）——这反映了不同的业务规则。例如"1个机构至少有1个门店"还是"可以没有门店"
  - 问题标志：所有一对多关系都标注为"1:*"而未区分下限是0还是1
  - 参考依据：《决胜B端》6.3.1节——ER图2"1..*的含义，至少要对应1个门店"

- [ ] **关系链的完整性**
  - 检查内容：关键实体之间的关系链是否完整，是否支撑核心业务场景
  - 优质标准：沿核心业务流程走查，确认每个环节涉及的实体都有正确的关系链连接。例如"客户→订单→商品→发货单→物流单"的完整链路
  - 问题标志：关系链存在断裂——两个在业务上相关的实体在ER图中没有直接或间接关系
  - 参考依据：《决胜B端》5.1节——ER模型必须支撑各种业务诉求

### 7.3 核心实体详细设计

- [ ] **实体属性的完整性**
  - 检查内容：核心实体是否列出了关键属性（字段）
  - 优质标准：至少列出实体的标识字段（ID）、核心业务字段、状态字段、时间戳字段（创建时间/更新时间），关键字段说明了数据类型和约束条件
  - 问题标志：实体只有名称没有属性，或关键业务字段遗漏
  - 参考依据：《决胜B端》6.3.1节ER图格式——方框第二行描述对象中的数据字段

- [ ] **状态设计的完整性**
  - 检查内容：有状态变化的实体（如订单、审批单、工单）是否有完整的状态机设计
  - 优质标准：提供了状态机图（State Machine Diagram），列出所有状态及状态间的转换条件；状态命名清晰无歧义；无"死状态"（无法流转到其他状态的死胡同）
  - 问题标志：有状态实体但无状态机图，或状态流转不完整（如"已提交"后缺少审批的状态变化路径）
  - 参考依据：《决胜B端》6.3节UML图——状态机图是产品经理常用的UML图之一

- [ ] **字段级别的规则描述**
  - 检查内容：PRD是否按字段描述了数据规则（默认值、校验规则、联动规则）
  - 优质标准：列表页和表单页的字段有完整的规则描述——字段名称、默认值、字段类型、是否必输、开放修改、长度限制、校验规则等
  - 问题标志：字段描述缺失，或只有字段名称而无规则
  - 参考依据：《决胜B端》PRD模板10.2节——查询条件和列表字段的结构化表格格式

### 7.4 数据治理考量

- [ ] **数据一致性保障**
  - 检查内容：是否考虑了跨模块、跨系统的数据一致性
  - 优质标准：识别了数据主源（谁是数据的Owner），定义了数据同步策略，说明了数据冲突的处理方式
  - 问题标志：同一数据在多处维护但未说明如何保持一致
  - 参考依据：《决胜B端》1.3.4节MDM主数据管理——"有且只有一套相关的资料存储系统"

- [ ] **数据模型的前瞻性**
  - 检查内容：数据模型是否考虑了未来可能的业务变化
  - 优质标准：关键实体的设计预留了扩展空间（如用可扩展的属性字段而非硬编码），对可能的业务演变（如从单一客户类型到多客户类型）有预判
  - 问题标志：数据模型只满足当前需求，一旦业务变化就需要大规模重构
  - 参考依据：《决胜B端》5.1节——"如果一上来就按最简单的方案设计，未来想切换到理想模型，研发投入会非常巨大"

- [ ] **历史数据和数据归档**
  - 检查内容：是否考虑了历史数据的处理和数据归档策略
  - 优质标准：说明了数据保留周期、归档策略、历史数据查询需求；对于数据量大的实体，有分表/分库的预案
  - 问题标志：未考虑数据增长带来的问题，或需要保留的历史数据会被覆盖或删除
  - 参考依据：《决胜B端》数据建模最佳实践

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| 无ER图 | 跳过建模直接画页面 | 先画ER图，再设计页面和交互 |
| 实体过多过碎 | 未做合理抽象 | 识别相似实体进行抽象合并 |
| 实体过于抽象 | 过度工程化设计 | 确保抽象不影响业务语义理解 |
| 关系类型错误 | 业务理解不够深入 | 用具体业务场景验证每个关系 |
| 无状态机图 | 遗漏状态管理设计 | 对所有有状态的实体补充状态机图 |
| 数据模型改一次架构崩一次 | 初始建模不够前瞻 | 先做加法考虑所有场景，再做减法选择合适方案 |
| 同一数据多处维护 | 缺乏主数据管理意识 | 确定数据主源，建立数据同步机制 |

# 交互设计质量检查

> Supporting reference for the `check-prd` skill. 在执行交互设计质量检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

B端产品的用户体验长期被忽视，但随着行业成熟，体验已成为产品竞争力的重要组成。《决胜体验设计》系统性地阐述了B端产品的体验设计方法论，从设计价值观到具体的交互范式，提供了完整的设计指导框架。

**核心理论框架：**

1. **体验设计四层模型**（《决胜体验设计》1.1节）：界面设计UI → 用户体验设计UX → 客户体验设计CX → 服务设计SD。四个层次逐层递进，B端产品经理至少要掌握UI和UX两层。

2. **设计价值观体系**（《决胜体验设计》第6章）：设计价值观 → 设计原则 → 设计范式。价值观是交互设计的灵魂，是对具体设计产出的评判标准。

3. **尼尔森十大可用性原则**（《决胜体验设计》6.2节引用）：系统状态可见性、系统与现实世界的匹配、用户控制与自由、一致性与标准化、错误预防、识别而非回忆、灵活高效、美观简洁、帮助用户识别诊断和恢复错误、帮助文档。

4. **用户体验五要素**（Jesse James Garrett，《决胜体验设计》引用）：战略层 → 范围层 → 结构层 → 框架层 → 表现层，从抽象到具体的五个层次。

5. **消息系统设计**（《决胜体验设计》8.4节）：推送人、推送对象、推送内容、推送渠道四要素；五个推送渠道（站内信、移动端Push、第三方APP、邮件、短信）；消息分级管理。

6. **新用户引导设计**（《决胜体验设计》8.3节）：Onboarding上手引导旅程设计，包括分步骤引导、功能发现、首次使用体验优化。

## 检查清单

### 9.1 设计价值观与原则

- [ ] **设计价值观的一致性**
  - 检查内容：PRD中的交互设计是否体现了一致的设计价值观
  - 优质标准：整个产品的交互风格统一——术语一致、操作范式一致、反馈方式一致、视觉风格一致。如果有设计系统或组件库引用，明确说明
  - 问题标志：同一产品中不同模块的交互风格迥异，似乎由不同人独立设计
  - 参考依据：《决胜体验设计》第6章——优秀的产品都应该有明确的设计价值观

- [ ] **一致性与标准化**（尼尔森原则）
  - 检查内容：相同操作在不同页面是否保持一致的交互方式
  - 优质标准：列表页的查询、筛选、排序操作在所有模块保持一致；表单的校验、保存、取消逻辑统一；操作反馈（Toast/弹窗/页面跳转）在相似场景下保持一致
  - 问题标志：同样的操作在不同页面有不同的交互方式（如有的列表支持搜索有的不支持，有的删除有二次确认有的直接删除）
  - 参考依据：尼尔森第四原则——一致性与标准化

### 9.2 信息架构与导航

- [ ] **信息架构的合理性**
  - 检查内容：产品的信息架构是否符合用户的心智模型
  - 优质标准：功能菜单按用户的业务逻辑组织（如按业务阶段、按角色职能），而非按技术模块组织；一级导航不超过7个（符合米勒法则7±2）；高频功能在醒目位置
  - 问题标志：菜单组织按技术模块而非用户视角；一级导航超过10个；高频功能隐藏在三级菜单中
  - 参考依据：《决胜体验设计》第5章信息架构——"让用户理解软件，找到所需"

- [ ] **导航设计的清晰度**
  - 检查内容：用户是否总能知道自己"在哪里"和"能去哪里"
  - 优质标准：有面包屑导航或层级标题指示当前位置；菜单的选中状态明确；全局导航在所有页面可用；提供搜索或快捷入口帮助用户快速跳转
  - 问题标志：用户点击几次后不知道自己在哪；返回路径不清晰；菜单选中状态缺失
  - 参考依据：尼尔森第一原则——系统状态可见性

### 9.3 核心交互设计

- [ ] **列表页设计规范**
  - 检查内容：列表页是否包含标准的交互元素
  - 优质标准：有查询条件区（支持简单查询和高级查询切换）；有列表展示区（支持排序、分页、列定制）；有批量操作区；每行有明确的操作入口（查看/编辑/删除等）；空状态有友好提示
  - 问题标志：列表页缺少查询功能；无分页或分页不合理；无空状态设计
  - 参考依据：《决胜B端PRD模板》10.2节列表页标准描述格式

- [ ] **表单设计规范**
  - 检查内容：表单页是否遵循良好的表单设计原则
  - 优质标准：必填项有明确标识（如红色星号）；字段有合理的排列顺序（重要字段在前）；有实时校验和错误提示；支持暂存/草稿；长表单有分步骤或分组设计；提交后有明确反馈
  - 问题标志：所有字段平铺无分组；无必填标识；校验信息不明确；长表单不支持暂存
  - 参考依据：《决胜体验设计》1.1.1节表单填写体验案例——百字段表单不支持暂存是典型反面案例

- [ ] **操作反馈设计**
  - 检查内容：用户操作后是否有明确的反馈
  - 优质标准：每个操作都有反馈（加载中状态、成功提示、失败原因说明）；关键操作有二次确认（如删除）；异步操作有进度指示；错误信息具体且可操作
  - 问题标志：操作后无反馈（用户不知道是否成功）；错误提示不具体（如只说"操作失败"不说原因）
  - 参考依据：尼尔森第九原则——帮助用户识别、诊断和恢复错误

- [ ] **错误预防设计**
  - 检查内容：是否有机制预防用户犯错
  - 优质标准：危险操作（删除、重置、批量修改）有二次确认；有输入校验和约束（如日期选择器限制不合理的日期范围）；有操作历史和撤销机制；自动保存草稿防止数据丢失
  - 问题标志：危险操作无确认直接执行；无输入约束导致用户容易填错
  - 参考依据：尼尔森第五原则——错误预防优于错误恢复

### 9.4 消息与通知设计

- [ ] **消息系统设计**
  - 检查内容：是否设计了系统的消息通知体系
  - 优质标准：
    - 定义了消息类型和分级（紧急/重要/普通/低优先级）
    - 定义了推送渠道策略（哪类消息走站内信、哪类走Push、哪类走邮件/短信）
    - 消息内容有概要信息而非空洞标题
    - 消息不滥用，有合理的发送频率控制
  - 问题标志：无消息设计；或消息被滥用（什么都推送）；或消息不分级（重要消息被淹没）；或消息文案不描述概要信息
  - 参考依据：《决胜体验设计》8.4节消息系统设计——推送人、推送对象、推送内容、推送渠道四要素；站内信、移动端Push、第三方IM、短信/电话、邮件五种渠道

- [ ] **新用户引导（Onboarding）**
  - 检查内容：如果是SaaS产品或新系统，是否设计了新用户上手引导
  - 优质标准：有分步骤引导流程帮助新用户完成首次关键操作；引导可跳过且可重新触发；引导结束后用户能独立使用核心功能
  - 问题标志：新用户进入空白系统无任何引导，不知道从何开始
  - 参考依据：《决胜体验设计》8.3节新用户引导设计——Salesforce等SaaS产品的Onboarding最佳实践

### 9.5 移动端体验（如适用）

- [ ] **移动端适配设计**
  - 检查内容：如果产品包含移动端，是否有针对移动场景的交互设计
  - 优质标准：移动端不是PC端的简单缩小版，而是根据移动场景重新设计（如支持手势操作、适配单手操作、考虑弱网环境、设计离线能力）
  - 问题标志：移动端设计照搬PC端布局，未考虑移动场景特性
  - 参考依据：《决胜B端》1.1.4节移动互联网时代——"如何在一块小小的屏幕内设计体验良好的App"

### 9.6 组件级交互设计分析（必做专项）

> 本节是本维度最重要的分析，**不能跳过**。前面 9.1-9.5 是合规检查，本节是深度评审。

**执行方法：**

第一步：从 PRD 中提取所有界面交互点，建立清单。包括但不限于：
- 所有列表页、详情页、表单页（新建/编辑）
- 所有弹窗、抽屉、浮层、Toast
- 所有重要操作（提交、删除、审批、导出、批量操作等）
- 所有状态切换（Tab切换、步骤切换、状态流转）

第二步：对每个交互点，评估以下组件选择决策：

**① 弹出层类型选择**（最常见的错误来源）
- **模态弹窗（Modal）**：适合需要用户立即处理、不能忽略、且操作简单的场景（如删除确认、提交确认）。不适合：内容复杂的表单、需要对照列表数据操作、用户可能频繁打开的场景
- **抽屉（Drawer/Slide-over）**：适合需要对照背景数据的详情查看或编辑、操作较复杂但不需要强制打断工作流的场景（如查看订单详情时参考列表）
- **跳转到新页面**：适合表单字段多（>10项）、操作复杂、需要完整视野的场景（如复杂合同录入）
- **非模态提示（Toast/Snackbar）**：适合操作成功/失败反馈，不应用模态弹窗来反馈普通操作结果

**② 表单交互模式**
- 长表单（>15字段）：应分步骤（Wizard）或分组折叠，而非平铺
- 频繁编辑的字段：考虑行内编辑（Inline Edit）代替跳转编辑页
- 关联选择：字段值多（>20项）应用搜索选择（Select + Search）而非普通下拉；存在强关联关系的字段（如省市区三级联动）是否设计了联动逻辑

**③ 列表交互模式**
- 详情查看方式：点击行→抽屉（推荐，保留列表上下文）vs 点击行→跳转详情页（适合详情内容极多），PRD 选择是否与场景匹配
- 批量操作：是否设计了批量选中后的操作栏（浮动工具栏 vs 顶部操作区）
- 空状态：是否设计了空列表引导（带操作按钮的空状态 vs 纯文字提示）

**④ 操作反馈类型匹配**
- 轻量操作成功（如点赞、收藏）：Toast 足够，不需要弹窗
- 重要操作成功（如提交审批、下单）：Toast + 页面状态更新，或带操作入口的通知（如"审批已提交，查看进度→"）
- 危险操作（删除、重置）：必须有模态二次确认，且确认按钮为危险色（红色），而非直接执行
- 异步操作（导入、导出、批量处理）：需要进度指示 + 完成通知，不应让用户面对白屏等待

**第三步：输出组件级分析结论**，格式如下：

```
【组件级问题】PRD第X节 - [功能名]
  当前设计：[描述PRD中的交互方式]
  问题：[为什么不合适]
  建议改为：[具体替代方案]
  原因：[一句话说明更合适的理由]
```

**注意：** 如果 PRD 对某个界面的交互方式描述不详细，也要指出"第X节未说明XX的交互方式，建议补充说明是弹窗还是跳转页，会直接影响研发实现"。

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| 不同模块交互风格不统一 | 缺少设计价值观和设计系统 | 建立统一的设计原则和组件库 |
| 菜单层级过深 | 按技术模块而非用户视角组织 | 按业务流程或用户角色重新组织 |
| 表单体验差 | 未遵循表单设计最佳实践 | 添加分组、暂存、实时校验、进度指示 |
| 操作后无反馈 | 遗漏反馈设计 | 为每个操作设计加载、成功、失败三种反馈 |
| 消息推送被用户忽视 | 消息滥用不分级 | 建立消息分级和渠道策略 |
| 新用户不知如何开始 | 缺少Onboarding设计 | 设计分步骤引导和空状态提示 |

# 商业分析深度检查

> Supporting reference for the `check-prd` skill. 在执行商业分析深度检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

> **适用范围：本维度仅适用于商业化产品（对外销售/运营的产品）。自研内部系统不适用，跳过此维度，改为在10.3节做ROI分析即可。**

B端产品最终要为企业创造商业价值。《决胜B端》强调产品经理必须"定义问题、解决问题、为价值交付负责"，商业分析是确保产品方向正确的关键。特别是SaaS产品，商业模式的可行性直接决定产品的生死。

**核心理论框架：**

1. **产业链分析框架**（《决胜B端》3.1节）：从宏观的产业链视角理解业务大盘——行业上下游关系、价值链分布、各环节的利润构成和核心玩家。

2. **市场结构分析模型**（《决胜B端》3.1节）：金字塔形、枣核形、倒三角形、哑铃形四种市场结构，决定了SaaS产品的可行性和定价策略。

3. **SaaS商业模式验证**（《决胜B端》第11-12章）：SaaS产品的核心指标包括CAC（客户获取成本）、LTV（客户生命周期价值）、MRR/ARR（月/年经常性收入）、Churn Rate（流失率）。LTV>3倍CAC是SaaS健康的基本门槛。

4. **PRD商业分析框架**（《决胜B端PRD模板》第3节）：目标市场与客户分析（目标市场、市场规模、市场特征、发展趋势、客户画像、客户痛点、卖点提炼）+ 竞品分析（商业模式、目标客户、运营推广策略、市场份额、产品分析、核心价值）。

5. **需求交付效率分析**（《决胜B端》第10章）：通过需求交付周期、PRD编写周期、研发等待周期等指标衡量产品团队效率和价值交付能力。

## 检查清单

### 10.1 市场与行业分析

- [ ] **产业链定位**
  - 检查内容：是否分析了产品在产业链中的位置和价值
  - 优质标准：绘制了目标行业的产业链图，标注了各环节的核心玩家和利润分布，明确了本产品切入的环节和价值创造点
  - 问题标志：未分析产业链，不清楚产品在行业生态中的位置
  - 参考依据：《决胜B端》3.1节产业链分析

- [ ] **市场规模估算**
  - 检查内容：是否估算了目标市场的规模（TAM/SAM/SOM）
  - 优质标准：用自上而下或自下而上的方法估算了市场规模（如"全国约有2000万个门店，按每家5000元/年计算，总市场规模约1000亿"），数据来源有引用
  - 问题标志：无市场规模分析，或数据凭空捏造
  - 参考依据：《决胜B端PRD模板》第3节——"市场规模：产品切入市场的规模"

- [ ] **市场特征与趋势**
  - 检查内容：是否分析了目标市场的竞争格局和发展趋势
  - 优质标准：描述了市场当前的竞争态势（如"两强竞争+多方蚕食"）、进入门槛、关键成功因素和发展趋势
  - 问题标志：市场描述过于乐观（如"蓝海市场""无竞争"），或缺乏具体竞争态势分析
  - 参考依据：《决胜B端PRD模板》第3节——市场特征、发展趋势

### 10.2 商业模式分析

- [ ] **收入模式清晰度**
  - 检查内容：是否明确了产品的收入模式
  - 优质标准：说明了产品如何赚钱（订阅制/按量计费/一次性许可/交易抽成/增值服务等），有基本的定价逻辑和价格区间参考
  - 问题标志：商业化产品但无收入模式说明；或自研系统但未分析成本收益比
  - 参考依据：《决胜B端》SaaS商业模式分析

- [ ] **客户获取策略**
  - 检查内容：是否说明了如何获取目标客户
  - 优质标准：描述了获客渠道（直销/渠道/线上/口碑）、获客成本预估、转化漏斗设计；对于SaaS产品，有免费试用/Freemium/PLG等获客策略的考量
  - 问题标志：只设计了产品功能，完全未考虑如何让客户知道并购买
  - 参考依据：《决胜B端PRD模板》第3节——竞品运营、推广、营销策略

- [ ] **SaaS关键指标预估**（SaaS产品必查）
  - 检查内容：是否分析了SaaS核心商业指标
  - 优质标准：预估了CAC（客户获取成本）、LTV（客户生命周期价值）、目标续约率/流失率、回本周期；LTV/CAC比值>3
  - 问题标志：SaaS产品未分析LTV/CAC，不清楚商业模式是否可持续
  - 参考依据：《决胜B端》SaaS商业模式——LTV>3xCAC是基本健康线

### 10.3 投入产出分析

- [ ] **项目投入估算**
  - 检查内容：是否估算了项目的研发投入（人力、时间、资金）
  - 优质标准：有粗略的工期估算和资源需求（如"预计需要5名研发，3个月完成V1.0"），标注了关键里程碑
  - 问题标志：完全没有投入估算，无法评估项目的可行性
  - 参考依据：《决胜B端》第10章项目管理——需求交付周期和工时利用率分析

- [ ] **收益预期与ROI**
  - 检查内容：是否量化了项目的预期收益，并计算了投入产出比
  - 优质标准：
    - 商业化产品：预估了收入增长或成本节约的具体金额
    - 自研系统：预估了效率提升幅度（如"预计减少50%的人工操作时间"）或风险降低程度
    - ROI=（收益-投入）/投入，有明确的回本预期
  - 问题标志：无收益预期，或收益预期无法量化
  - 参考依据：《决胜B端PRD模板》第4节——"具体的量化的项目目标"和"收益预期，需要量化"

- [ ] **机会成本考量**
  - 检查内容：是否考虑了做这个项目的机会成本——同样的资源投入其他项目是否更有价值
  - 优质标准：将本项目与其他备选项目对比，说明了为什么优先做本项目
  - 问题标志：项目决策缺乏优先级对比依据
  - 参考依据：B端产品资源管理最佳实践

### 10.4 数据驱动验证（商业化产品专项）

> 注：软件使用监控指标（用户/行为/业务/技术数据）适用于所有产品类型，已纳入 check-prd-14（维度14-运营方案与效果跟踪）的「上线跟踪与效果评估」中。本节仅检查商业化产品特有的售卖侧指标。

- [ ] **软件售卖分析指标**（仅适合商业化SaaS产品）
  - 检查内容：是否定义了衡量软件商业化情况的核心SaaS指标
  - 优质标准：覆盖以下两类数据：
    - **商业数据**：NDR（净收入留存率）、ARR（年度经常性收入）、MRR（月度经常性收入）、LTV（客户生命周期价值）、CAC（客户获取成本）；LTV/CAC > 3为健康基准线
    - **客户数据**：客户来源（自然流量/转介绍/采购流量），客户类型（付费/免费；活跃/流失/唤醒），客户级别（VIP客户/普通客户）
  - 问题标志：SaaS产品未关注NDR和LTV/CAC，不知道商业模式是否可持续；缺少客户来源分析无法优化获客
  - 参考依据：《决胜B端》SaaS商业模式分析；《决胜B端》产品监控的指标体系设计

- [ ] **数据采集方案**
  - 检查内容：是否设计了数据采集方案来支撑后续的数据分析
  - 优质标准：定义了需要埋点的核心页面和操作、数据采集频率、数据分析维度；与PRD第11节数据埋点章节呼应
  - 问题标志：定义了指标但不知道数据从哪来、如何采集
  - 参考依据：《决胜B端PRD模板》第11节数据埋点

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| 无市场分析直接做功能 | 产品经理缺乏商业意识 | 先做市场和商业分析，再做功能设计 |
| 市场规模过于乐观 | 数据来源不可靠或计算逻辑有误 | 用可信数据源，自下而上交叉验证 |
| SaaS不算LTV/CAC | 不理解SaaS商业模式核心 | 补充SaaS核心指标分析 |
| 无投入估算 | 产品和技术沟通不充分 | 与技术负责人一起粗估工期和资源 |
| 收益无法量化 | 缺乏度量体系设计 | 定义可度量的成功指标，设计数据采集方案 |

# MVP策略与演进蓝图检查

> Supporting reference for the `check-prd` skill. 在执行 MVP 策略与演进蓝图检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

B端产品建设是一个长期过程，不可能一步到位。MVP（最小可行产品）策略和产品演进蓝图决定了产品如何从0到1、从1到N逐步成长。《决胜B端》强调产品建设应"先做加法再做减法"——先充分理解所有业务场景，再有策略地裁剪出MVP。

**核心理论框架：**

1. **MVP设计原则**（《决胜B端》第2章和第4章）：MVP不是"最简单的版本"，而是能够验证核心假设、支撑核心业务闭环的最小功能集。产品定位明确后，需要规划"功能模块与演进蓝图"。

2. **先做加法再做减法**（《决胜B端》5.1节）："我们做产品设计，重要的是能够想清楚所有的业务场景和问题，然后基于研发资源和实现周期，做出一个综合评估后的合理方案"。

3. **种子客户策略**（《决胜B端》3.1节）：商业化产品在MVP阶段需要选择"有代表性的标杆客户"进行充分调研和验证，MVP版本应围绕种子客户的核心需求刻画。

4. **产品化策略**（《决胜B端》第8章）：从项目制到产品化的演进路径——先解决单一客户问题，再抽象为标准化产品，最后建立配置化/插件化的产品平台。

5. **迭代与版本管理**（《决胜B端》第10章）：通过需求优先级管理、版本规划、需求池管理等手段持续迭代产品。

## 检查清单

### 11.1 MVP范围定义

- [ ] **MVP功能范围的明确性**
  - 检查内容：是否明确定义了MVP版本包含哪些功能、不包含哪些功能
  - 优质标准：列出MVP的功能清单，对每个功能说明"为什么在MVP中"（必要性）或"为什么不在MVP中"（延后理由）；MVP功能构成完整的业务闭环
  - 问题标志："什么都要做"没有裁剪；或MVP功能不构成闭环（如有下单但没有支付，有创建但没有查询）
  - 参考依据：《决胜B端》4.3节功能模块与演进蓝图

- [ ] **MVP的核心假设验证**
  - 检查内容：MVP是否围绕核心假设设计，上线后能验证哪些关键假设
  - 优质标准：明确列出MVP要验证的核心假设（如"目标客户愿意为X功能付费""Y流程能替代现有Excel方案"），以及如何通过数据验证这些假设
  - 问题标志：MVP设计仅基于功能裁剪而非假设验证导向
  - 参考依据：《决胜B端》产品建设思维——产品经理必须"为价值交付负责"

- [ ] **MVP的业务闭环完整性**
  - 检查内容：MVP是否覆盖了核心业务流程的完整闭环
  - 优质标准：沿核心业务流程走查，每个关键环节都有对应功能（即使是简化版本）；关键的数据流转不中断
  - 问题标志：MVP只有业务流程的前半段（如只有创建无管理，只有录入无报表），用户无法基于MVP完成完整业务操作
  - 参考依据：《决胜B端》产品建设流程——先梳理核心业务流程，再裁剪为MVP

### 11.2 演进蓝图规划

- [ ] **版本演进路线图**
  - 检查内容：是否规划了从MVP到成熟版本的演进路线图
  - 优质标准：有3-4个阶段的版本规划（如V1.0核心流程→V1.5体验优化→V2.0扩展功能→V3.0平台化），每个阶段有明确的功能范围和目标
  - 问题标志：只有V1.0设计，完全没有后续演进考虑；或规划过于远大（一上来就画3年蓝图但不务实）
  - 参考依据：《决胜B端》第4章产品整体方案——"功能模块与演进蓝图"

- [ ] **功能优先级矩阵**
  - 检查内容：非MVP功能是否有优先级排序和分期计划
  - 优质标准：使用优先级框架（如价值/成本矩阵、KANO模型、ICE评分）对功能进行优先级排序，高优先级功能在早期版本实现
  - 问题标志：所有功能同等重要无优先级，或优先级排序缺乏依据
  - 参考依据：《决胜体验设计》引用KANO模型——区分基本型需求、期望型需求、兴奋型需求

- [ ] **架构预留与技术债控制**
  - 检查内容：MVP是否在数据模型和架构层面为后续版本预留了扩展空间
  - 优质标准：MVP的数据模型虽然功能精简但结构完整，不会因为后续版本需要大规模重构数据库；关键扩展点有明确标注
  - 问题标志：MVP采用"临时方案"实现，明确会在后续版本全面重构
  - 参考依据：《决胜B端》5.1节——"一旦表结构定型，以后再想修改，是非常麻烦的事情"

### 11.3 种子客户策略

- [ ] **种子客户选择**（商业化产品必查）
  - 检查内容：是否选择了合适的种子客户来验证MVP
  - 优质标准：种子客户有代表性（代表目标客群的典型特征）、有配合意愿、痛点明确；选择了1-3个种子客户
  - 问题标志：无种子客户计划，打算MVP做完直接面向市场推广
  - 参考依据：《决胜B端》3.1节——"针对一个有代表性的标杆客户进行充分调研，透彻理解业务，并刻画出MVP版本"

- [ ] **从项目到产品的演进策略**（商业化产品必查）
  - 检查内容：是否规划了从服务单一客户（项目制）到标准化产品的演进路径
  - 优质标准：明确了第一阶段为种子客户做定制化实施，第二阶段提炼共性需求标准化，第三阶段建立配置化平台的演进策略
  - 问题标志：商业化产品但始终按单一客户定制化方式运作，无产品化计划
  - 参考依据：《决胜B端》第8章产品化——标准化和个性化之间的平衡

### 11.4 上线与运营准备

- [ ] **上线计划**
  - 检查内容：是否有MVP的上线计划，包括灰度策略和回滚方案
  - 优质标准：定义了上线范围（全量/灰度/AB测试）、上线时间表、数据迁移方案、回滚计划、上线后的监控指标
  - 问题标志：无上线计划，或计划"一次性全量上线"而无灰度策略
  - 参考依据：B端产品上线最佳实践

- [ ] **培训与推广计划**
  - 检查内容：是否规划了用户培训和系统推广方案
  - 优质标准：有用户培训方案（培训材料、培训方式、培训时间表）；有推广方案（如何让用户从旧系统/旧方式切换到新系统）
  - 问题标志：系统做好了但没人用，因为缺乏培训和推广
  - 参考依据：《决胜B端PRD模板》第13节运营计划——"需要执行的推广计划、部署计划、执行计划"

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| MVP什么都要做 | 未做功能裁剪，缺乏聚焦 | 以核心假设验证为导向裁剪功能 |
| MVP功能不构成闭环 | 裁剪时只考虑功能复杂度不考虑业务完整性 | 沿核心流程走查确保闭环 |
| 无演进蓝图 | 只关注当前版本缺乏长期规划 | 规划3-4个阶段的版本路线图 |
| 数据模型需要频繁重构 | MVP建模过于简化 | MVP阶段就做完整的数据建模，功能可精简但模型要全面 |
| 做了没人用 | 缺少推广和培训规划 | 在PRD中加入运营和培训计划 |
| 一直做项目制无法标准化 | 无产品化策略 | 明确从项目到产品的三阶段演进 |

# 运营方案与效果跟踪检查

> Supporting reference for the `check-prd` skill. 在执行运营方案与效果跟踪检查时加载；单独使用时请先提供待检查的 PRD 内容。

## 维度说明

B端产品的成败不只取决于功能设计质量，更取决于能否落地运营、持续改善、真正被用户接受。《决胜B端》明确指出，产品效果受多因素影响：员工能力、业务流程合理性、管理制度——因此产品方案必须配套完整的运营方案才能形成闭环。

**核心理论框架：**

1. **产品运营体系五大类**（《决胜B端》产品运营体系设计）：
   - 需求收集与信息同步
   - 功能宣传与推广
   - 用户服务与支持
   - 上线跟踪与效果评估
   - 业务复盘与影响力构建

2. **产品监控指标体系**（《决胜B端》产品监控的指标体系设计）：覆盖用户数据、行为数据、业务数据、技术数据四类，自研系统和商业化产品都需要，差异仅在是否需要商业化指标。

3. **运营计划要求**（《决胜B端PRD模板》第13节）：需要新建的运营流程、业务流程、客服处理流程；需要执行的推广计划、部署计划、执行计划。

4. **数据驱动运营**（《决胜B端》第10章）：通过行为漏斗、页面转化率、用户留存等数据持续指导产品迭代，而非仅依赖主观判断。

## 检查清单

### 14.1 需求收集与信息同步

- [ ] **需求提报机制**
  - 检查内容：是否建立了统一的需求收集入口和周期性沟通机制
  - 优质标准：统一收口（避免多渠道散乱），定期需求沟通会（双周/月），有结构化的原始需求采集表，让业务方知道需求在哪个阶段
  - 问题标志：需求通过微信零散传达，无统一入口；业务方不知道自己提的需求是否会做、什么时候做
  - 参考依据：《决胜B端》产品运营体系设计——需求收集与信息同步

- [ ] **需求进展同步机制**
  - 检查内容：是否有持续向业务方同步需求进展的机制
  - 优质标准：信息同步表（日报表/周报表），定期沟通会，让业务方知道需求在哪个阶段
  - 问题标志：业务方反映"不知道我的需求做没做""不知道什么时候上线"
  - 参考依据：《决胜B端》产品运营体系设计

### 14.2 功能宣传与推广

- [ ] **产品发布节奏**
  - 检查内容：是否设计了固定的产品发布节奏和宣传方式
  - 优质标准：固定的发布周期（双周/月），让用户有预期；固定宣传方案（邮件/长图海报/短视频），根据受众选择合适渠道
  - 问题标志：功能上线后不通知用户，或仅发一封系统邮件；用户不知道新功能的存在和用法
  - 参考依据：《决胜B端》产品运营体系设计——功能宣传与推广

- [ ] **培训与推广落地**
  - 检查内容：新功能上线是否有配套的培训和使用激励方案
  - 优质标准：培训答疑（线上/线下），竞赛形式促进使用（技能大赛/知识大赛/最佳案例评选）
  - 问题标志：功能上线后用户不会用、不想用；缺少引导新用户了解功能价值的方案
  - 参考依据：《决胜B端》产品运营体系设计

### 14.3 用户服务与支持

- [ ] **问题反馈机制**
  - 检查内容：是否设计了用户反馈和问题响应的完整链路
  - 优质标准：明确受理入口（邮箱/系统意见箱），有问题受理流程和SLA（问题等级/响应速度/排班机制）
  - 问题标志：用户遇到问题不知道找谁；反馈了没有回音；问题处理无优先级无SLA
  - 参考依据：《决胜B端PRD模板》第13节——"客服处理流程"

- [ ] **意见激励机制**
  - 检查内容：是否有鼓励用户提意见并给予反馈的机制
  - 优质标准：意见收集入口，意见采纳后有反馈（物质激励/精神激励），让用户感受到被重视
  - 问题标志：用户提意见如石沉大海，久而久之不再反馈，形成信息孤岛
  - 参考依据：《决胜B端》产品运营体系设计

### 14.4 上线跟踪与效果评估

上线跟踪是产品持续改进的基础。自研系统和商业化产品都需要，差异在于是否还需要商业化指标。

- [ ] **跟踪责任与周期**
  - 检查内容：是否明确了谁负责跟踪、多长周期评估一次
  - 优质标准：明确跟踪负责人，有固定评估周期（周/月/季度），每个重要功能有专属的结果数据和使用数据追踪
  - 问题标志：产品上线后没有效果评估；无人负责跟踪
  - 参考依据：《决胜B端》产品监控的指标体系设计

- [ ] **用户数据与行为数据**
  - 检查内容：是否定义了观察用户状态和使用行为的指标
  - 优质标准：覆盖以下维度：
    - **用户数据**：用户类型（新用户/老用户），用户行为状态（活跃用户/流失用户/沉睡用户）
    - **行为数据**：活跃度（PV、UV、访问深度、访问时长），访问路径（行为漏斗、页面转化率），质量（弹出率、退出率、留存率）
  - 问题标志：只有结果指标（如销售额），没有行为数据；出了问题无法诊断原因
  - 参考依据：《决胜B端》产品监控的指标体系设计

- [ ] **业务数据与技术数据**
  - 检查内容：是否定义了衡量业务效果和系统健康的指标
  - 优质标准：覆盖以下维度：
    - **业务数据**：结果数据（GMV/销售额/转化率/满意度），过程数据（拜访量/通话时长/货损率等业务过程指标）
    - **技术数据**：接口调用（API调用次数），系统性能（并发量、流量、错误率）
  - 问题标志：指标全是结果指标，缺过程数据；技术健康无人监控
  - 参考依据：《决胜B端》产品监控的指标体系设计

### 14.5 业务复盘与影响力构建

- [ ] **定期复盘机制**
  - 检查内容：是否设计了定期业务复盘和拉通机制
  - 优质标准：定期复盘总结会（季度/半年/年度），业产研三方参与，优秀评选（优秀项目/优秀个人/最佳搭档），让参与者有成就感
  - 问题标志：只埋头做产品，缺少总结和展示；产品效果无人关注，改进缺乏方向
  - 参考依据：《决胜B端》产品运营体系设计——业务复盘与影响力构建

- [ ] **产研影响力构建**
  - 检查内容：是否有计划通过事件营销扩大产品和团队的影响力
  - 优质标准：通过事件营销扩大影响力（节日活动/周年纪念/黑客马拉松等），让产品团队在公司内部获得话语权
  - 问题标志：产品团队在公司内部缺乏话语权，投入资源做了产品却无人感知价值
  - 参考依据：《决胜B端》产品运营体系设计

### 14.6 上线准备：数据初始化与迁移

- [ ] **数据初始化与迁移方案**
  - 检查内容：新系统上线是否需要数据初始化或从旧系统迁移数据
  - 优质标准：定义了初始化数据的来源、格式、导入方式；如果有旧系统，定义了数据迁移方案（迁移范围、映射规则、数据清洗策略、迁移验证方法）
  - 问题标志：新系统需要历史数据但没有迁移方案，或上线后是空系统无初始数据，第一天打开是白板
  - 参考依据：B端产品上线准备工作

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| 新系统上线没人用 | 缺少运营和培训方案 | 补充运营流程、培训计划、推广方案 |
| 业务方不知道需求进展 | 缺需求同步机制 | 建立固定的需求沟通会和进展同步表 |
| 上线后不知道效果好不好 | 缺指标体系和跟踪负责人 | 定义完整指标体系，指定专人负责跟踪 |
| 用户遇问题无处反馈 | 缺问题受理流程和SLA | 建立反馈入口，制定分级响应机制 |
| 产品团队话语权弱 | 缺成果展示和影响力建设 | 设计定期复盘机制，通过事件营销建立存在感 |
| 新系统第一天打开是空的 | 未提前规划数据初始化 | 在PRD中明确数据迁移和初始化方案 |

# 异常处理与健壮性设计检查

> Supporting reference for the `check-prd` skill. 在执行异常处理与健壮性设计检查时加载；单独使用时请先提供待检查的 PRD 内容。
> **注意**：运营方案与效果跟踪已独立为维度14，本维度专注于系统健壮性和风险管理。

## 维度说明

B端产品的真正考验不在于正常流程能否跑通，而在于异常情况能否妥善处理。《决胜B端》明确指出，产品方案"必须思考全面，包括异常情况的处理"。数据安全和合规性是B端产品不可忽视的底线，而提前识别并管理风险能让产品少走弯路。

**核心理论框架：**

1. **异常处理方案**（《决胜B端PRD模板》10.3节）：明确要求覆盖"断网、断电、误操作、数据丢失等"异常场景，异常处理方案可以写在专门章节或分散在各功能描述中。

2. **业务规则五种类型**（《决胜B端PRD模板》10.2节）：事实、约束、触发条件、推论、计算——其中"约束"类型专门描述"对业务动作和用户操作限制的描述"，包含"不符合约束条件时的预期响应"。

3. **项目风险管理**（《决胜B端PRD模板》第7节）：识别假设、约束、产品风险、运营风险、技术风险，并制定应对方案。

## 检查清单

### 12.1 异常场景覆盖

- [ ] **技术异常处理**
  - 检查内容：是否考虑了常见的技术异常场景
  - 优质标准：覆盖以下异常场景并有处理方案：
    - 网络中断/弱网环境：离线缓存、自动重试、断点续传
    - 服务不可用：友好的错误页面、降级策略
    - 数据加载超时：超时提示、取消/重试选项
    - 并发冲突：乐观锁/悲观锁策略、冲突提示
    - 数据丢失/损坏：自动备份、数据恢复方案
  - 问题标志：完全未考虑技术异常，默认系统永远正常运行
  - 参考依据：《决胜B端PRD模板》10.3节——"断网、断电"等异常

- [ ] **业务异常处理**
  - 检查内容：是否考虑了业务层面的异常场景
  - 优质标准：覆盖以下异常场景：
    - 误操作处理：支持撤销、回退、或管理员干预修正
    - 数据录入错误：校验前置、修改记录、审计追踪
    - 业务规则冲突：冲突检测、提示、人工裁决
    - 超时未处理：自动提醒、超时升级、自动取消/关闭
    - 角色缺失：审批人不在线/离职时的替代处理（代理审批、升级审批）
  - 问题标志：只考虑了理想业务路径，未考虑操作失误和业务冲突
  - 参考依据：《决胜B端PRD模板》10.3节——"误操作、数据丢失等"

- [ ] **边界条件处理**
  - 检查内容：是否考虑了数据和操作的边界条件
  - 优质标准：覆盖以下边界条件：
    - 空数据状态：列表为空时的友好提示和引导
    - 大数据量：分页、懒加载、导出限制
    - 并发操作：同一数据被多人同时编辑时的处理
    - 跨时区/跨区域：时区显示、多语言、货币转换
    - 极端输入：超长文本、特殊字符、超大文件
  - 问题标志：未考虑边界情况，用户在特殊条件下可能遇到系统崩溃或数据错误
  - 参考依据：B端产品健壮性设计最佳实践

### 12.2 数据安全与审计

- [ ] **操作审计设计**
  - 检查内容：是否设计了关键操作的审计追踪
  - 优质标准：关键业务操作（创建、修改、删除、审批、导出）有操作日志记录，包含操作人、操作时间、操作内容（修改前后值）、操作IP；日志不可篡改
  - 问题标志：无操作审计设计，出问题时无法追溯是谁做了什么操作
  - 参考依据：B端产品合规性要求

- [ ] **数据备份与恢复**
  - 检查内容：是否说明了数据备份策略和灾难恢复方案
  - 优质标准：定义了备份频率（如每日全量+实时增量）、备份存储位置、数据恢复的RTO（恢复时间目标）和RPO（恢复点目标）
  - 问题标志：无备份策略，或依赖"开发自己想办法"
  - 参考依据：企业级产品数据安全要求

- [ ] **敏感数据处理**
  - 检查内容：是否识别并保护了敏感数据（个人信息、财务数据、商业机密等）
  - 优质标准：识别了敏感数据字段并定义了保护措施（脱敏显示、加密存储、访问控制、日志审计），符合相关法规要求
  - 问题标志：包含敏感数据但无保护措施（如手机号/身份证号明文展示）
  - 参考依据：数据安全法规和B端产品安全设计

### 12.3 风险识别与应对

- [ ] **产品风险识别**
  - 检查内容：是否识别了产品层面的风险并制定了应对方案
  - 优质标准：识别了需求理解偏差风险、设计方案缺陷风险、用户接受度风险等，每个风险有概率评估和应对策略
  - 问题标志：PRD中无风险分析章节
  - 参考依据：《决胜B端PRD模板》第7节——"产品风险、运营风险、技术风险及应对方案"

- [ ] **技术风险识别**
  - 检查内容：是否识别了技术层面的风险
  - 优质标准：识别了技术可行性风险（如某功能技术上难以实现）、性能瓶颈风险、第三方依赖风险、安全风险等
  - 问题标志：假设所有技术方案都可行，未做风险评估
  - 参考依据：《决胜B端PRD模板》第7节

- [ ] **待决事项管理**
  - 检查内容：是否记录和管理了所有待决事项
  - 优质标准：每个待决事项有编号、描述、负责人、预计决策日期；待决事项数量控制在合理范围（<10个）；关键待决事项有备选方案
  - 问题标志：明显有未决问题但未记录；待决事项过多说明PRD不成熟
  - 参考依据：《决胜B端PRD模板》第14节待决事项

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| 只有Happy Path | 未做异常场景推演 | 对每个核心流程做"如果出错/如果超时/如果冲突"推演 |
| 无操作审计 | 缺乏合规意识 | 为关键操作添加审计日志设计 |
| 敏感数据明文展示 | 未识别和保护敏感数据 | 识别敏感字段，设计脱敏方案和访问控制 |
| 风险无人管理 | 缺少风险管理章节 | 补充风险识别和应对方案 |
| 上线后系统频繁出问题 | 边界条件和异常场景未覆盖 | 系统性检查所有边界条件和异常场景 |

# AI功能设计质量检查

> Supporting reference for the `check-prd` skill. 在执行 AI 功能设计质量检查时加载；单独使用时请先提供待检查的 PRD 内容。
> **注意**：本维度仅在PRD涉及AI功能时进行评估。如果PRD不涉及AI功能，标记为"不适用"并跳过。

## 维度说明

随着大语言模型（LLM）的快速发展，越来越多的B端产品开始集成AI能力。《决胜体验设计》第9章系统性地阐述了AI时代B端产品的交互设计方法论，提出了"六脉神剑"框架来指导AI功能的设计。正确的AI功能设计能极大提升产品价值，错误的设计则可能导致用户困惑甚至业务风险。

**核心理论框架：**

1. **人机交互三个历史阶段**（《决胜体验设计》9.1节）：以机器为中心的设计（19世纪末-20世纪80年代）→ 以人为中心的设计（20世纪80年代-现在）→ 以结果为中心的设计（AI时代）。AI时代的核心转变是用户告诉AI想要什么，AI直接交付结果。

2. **确定性-容错率二维模型**（《决胜体验设计》9.2节）：以"确定性任务vs探索性任务"为横轴，"容错率高vs容错率低"为纵轴，形成四个象限，不同象限适用不同的AI交互模式。

3. **AI交互六脉神剑**（《决胜体验设计》9.2节）：
   - **第一剑：封装API**——将AI能力封装在传统GUI按钮后面，用户点击按钮触发AI，适用于确定性高、容错率高的场景（如一键生成摘要）
   - **第二剑：将CUI嵌入GUI**——在GUI界面中嵌入对话式交互，适用于容错率高的探索性任务（如AIGC生成图片、文案）
   - **第三剑：AI Agent侧边栏**——统一的智能体侧边栏，在任何页面可唤起，适用于需要跨页面、多步骤操作的场景
   - **第四剑：AI辅助填充**——AI自动填充表单、自动推荐选项，适用于确定性高但容错率低的场景
   - **第五剑：AI后台自动化**——AI在后台自动执行任务无需用户干预，适用于完全确定性的自动化场景
   - **第六剑：全CUI替代GUI**——纯对话式交互，用户完全通过自然语言操控系统，适用于AI编程等高度灵活的场景

4. **AI Agent概念**（《决胜体验设计》9.2.4节）：AI Agent是承载AI从思考到落地闭环的实体，各大SaaS厂商（Salesforce的Agentforce、Atlassian的Rovo、HubSpot的Breeze、Shopify的Sidekick）都采用了智能体侧边栏的设计模式。

5. **Copilot模式**（《决胜体验设计》引用）：AI作为用户的助手（副驾驶）存在，用户保持控制权，AI提供建议和辅助。

## 检查清单

### 13.1 AI功能定位

- [ ] **AI功能的价值主张**
  - 检查内容：AI功能要解决什么问题，为用户带来什么价值
  - 优质标准：明确说明AI功能的目标（如"帮助销售人员自动生成客户跟进建议，节省每日30分钟的分析时间"），而非为了"AI化"而加AI
  - 问题标志：AI功能缺乏明确的价值主张（如"我们也加个AI功能"），或AI功能与产品核心价值无关
  - 参考依据：《决胜体验设计》9.1节——AI的核心价值是"以结果为中心"，让用户直接获得结果

- [ ] **AI适用场景判断**
  - 检查内容：所选场景是否适合用AI来解决
  - 优质标准：使用确定性-容错率二维模型判断场景所在象限，并选择了匹配的交互模式；AI适合用于数据分析、内容生成、模式识别、自动化等场景
  - 问题标志：在不适合AI的场景使用AI（如用AI替代简单的下拉选择，或在容错率极低的场景用不可靠的AI做自动决策）
  - 参考依据：《决胜体验设计》9.2节确定性-容错率二维模型

### 13.2 AI交互设计

- [ ] **AI交互模式选择**
  - 检查内容：AI功能的交互模式是否与场景匹配
  - 优质标准：根据六脉神剑框架选择了正确的交互模式：
    - 确定性高+容错率高→封装API（一键按钮）
    - 探索性+容错率高→CUI嵌入GUI（对话框）
    - 跨页面多步骤→AI Agent侧边栏
    - 确定性高+容错率低→AI辅助填充（需人工确认）
    - 完全自动化→AI后台自动化
  - 问题标志：交互模式与场景不匹配（如用全CUI替代简单的一键操作，或在高风险场景用无确认的自动化）
  - 参考依据：《决胜体验设计》9.2节AI交互六脉神剑

- [ ] **人机协作边界**
  - 检查内容：AI和人之间的职责边界是否清晰
  - 优质标准：明确了哪些步骤由AI完成、哪些步骤需要人工确认、哪些步骤人工不可绕过；在关键决策点保留了人工干预的能力（Human-in-the-Loop）
  - 问题标志：AI自动做出关键业务决策但用户无法干预或修改；或AI功能需要过多人工干预失去了效率优势
  - 参考依据：《决胜体验设计》Copilot模式——AI作为助手，用户保持控制权

- [ ] **AI输出的可解释性**
  - 检查内容：AI的输出结果是否对用户有解释性
  - 优质标准：AI给出建议或结果时附带了推理依据（如"基于该客户近30天的浏览记录和购买历史，推荐推送XX产品"）；用户能理解AI为什么给出这个结果
  - 问题标志：AI给出结果但不解释原因（"黑盒"输出），用户无法判断结果是否可信
  - 参考依据：AI产品可信赖设计原则

### 13.3 AI可靠性设计

- [ ] **AI错误处理与回退**
  - 检查内容：当AI输出错误或不可用时，是否有回退方案
  - 优质标准：设计了AI不可用时的降级方案（如AI推荐不可用时回退到规则引擎或手动操作）；AI生成内容有人工修改和覆盖的能力；AI服务异常时用户仍能完成核心操作
  - 问题标志：核心业务流程完全依赖AI，AI不可用时业务中断
  - 参考依据：B端产品可靠性设计原则

- [ ] **AI输出质量保障**
  - 检查内容：是否有机制保障AI输出的质量
  - 优质标准：定义了AI输出的质量标准和评估方法；有异常输出的检测和过滤机制；用户可以对AI输出进行反馈（点赞/点踩）以持续优化
  - 问题标志：对AI输出质量无任何管控，直接展示给用户或执行
  - 参考依据：AI产品质量管理

- [ ] **数据隐私与安全**
  - 检查内容：AI功能是否考虑了数据隐私和安全
  - 优质标准：说明了AI处理的数据范围和类型；明确了数据是否传输到外部AI服务（如调用第三方LLM API）；敏感数据（客户信息、财务数据）是否做了脱敏处理后再传给AI
  - 问题标志：将敏感业务数据直接传给第三方AI服务而无脱敏和安全措施
  - 参考依据：数据安全和隐私保护要求

### 13.4 AI体验优化

- [ ] **AI响应时间与加载体验**
  - 检查内容：是否考虑了AI处理的响应时间和等待体验
  - 优质标准：对于耗时较长的AI操作（如生成长文档），设计了流式输出（typing效果）、进度指示、异步通知等交互方式；用户在等待时可以继续其他操作
  - 问题标志：AI操作需要较长时间但无任何等待反馈，用户不知道系统在做什么
  - 参考依据：尼尔森第一原则——系统状态可见性

- [ ] **AI功能的渐进式呈现**
  - 检查内容：AI功能是否采用了渐进式设计，而非强迫用户使用
  - 优质标准：AI功能作为可选的辅助能力存在，不强制用户使用；新用户有引导了解AI能力；高级用户可以深度使用AI功能
  - 问题标志：AI功能强制替代了原有操作方式，不给用户选择权
  - 参考依据：《决胜体验设计》AI设计原则——渐进式引入AI能力

### 13.5 AI功能监控指标设计

AI产品的监控指标不同于普通功能，需要额外关注AI本身的能力表现。PRD中应定义上线后如何衡量AI功能是否真正有效。

- [ ] **使用效果指标**
  - 检查内容：是否定义了衡量AI能力表现的质量指标
  - 优质标准：覆盖以下维度：
    - **任务解决率**：问题解决率（用户采纳AI方案的比例）；监控方法：日志分析 + 行为分析 + 满意度评分
    - **理解准确性**：意图识别准确率、实体抽取准确率；监控方法：每日自动化测试 + 人工抽检
    - **回复质量**：答非所问率、回复完整度评分；监控方法：NLU模型评估 + 用户反馈
    - **多轮对话能力**（对话型AI适用）：平均对话轮次、上下文理解断裂率；监控方法：会话链路分析
  - 问题标志：只监控AI功能的使用量，不监控AI输出是否真的准确有效
  - 参考依据：《决胜体验设计》AI产品监控指标体系

- [ ] **用户行为指标**
  - 检查内容：是否定义了观察用户如何使用AI功能的行为指标
  - 优质标准：覆盖以下维度：
    - **使用深度**：提问复杂度分布、高级功能使用率；说明：引导用户加深使用
    - **放弃行为**：中途退出率、转人工率及原因分类；说明：定位AI能力覆盖不足之处
    - **问题类型**：高频问题TOP20、未覆盖问题占比；说明：指导知识库扩展方向
  - 问题标志：不分析放弃和转人工行为，无法发现AI能力瓶颈
  - 参考依据：《决胜体验设计》AI产品监控指标体系

- [ ] **推广渗透指标**
  - 检查内容：是否定义了衡量AI功能在团队/组织内渗透程度的指标
  - 优质标准：覆盖以下维度：
    - **覆盖率**：员工实际使用率（有多少人真的在用）
    - **替代率**：AI处理占比 vs 人工处理占比（AI实际替代了多少人工工作量）
    - **依赖度**：关键业务流程中AI参与度（AI是否真正融入日常工作）
  - 问题标志：只看功能上线没人问，不知道员工是否真的在用；无法回答"AI上线后到底替代了多少人工"
  - 参考依据：《决胜体验设计》AI产品监控指标体系

## 常见问题诊断

| 问题现象 | 根本原因 | 改进方向 |
|---------|---------|---------|
| 为AI而AI | 缺乏明确的AI价值主张 | 先明确AI要解决的具体问题和带来的具体价值 |
| AI交互方式与场景不匹配 | 未使用确定性-容错率模型判断 | 用六脉神剑框架选择正确的交互模式 |
| AI做了关键决策用户无法干预 | 人机边界设计不当 | 在关键决策点保留Human-in-the-Loop |
| AI不可用时业务中断 | 过度依赖AI缺乏降级方案 | 设计AI不可用时的回退方案 |
| 敏感数据泄露给第三方AI | 未做数据安全评估 | 对AI使用的数据做脱敏和安全审查 |
| AI输出不可信用户不敢用 | 缺乏可解释性设计 | 为AI输出添加推理依据和置信度指示 |

# 重大风险项清单 + 常见问题对照表

> Supporting reference for the `check-prd` skill. 在所有维度检查完成后加载，用于识别重大风险项与高风险区域。

## 重大风险项清单

以下任何一项如果存在，建议在评审时重点讨论，推进前应有明确的解决方案。这些是产品设计的底线问题，影响系统的可用性或合规性。

### R1. 产品定位缺失
- **触发条件**：PRD通篇无法回答"这个产品是为谁解决什么问题"
- **判断标准**：阅读PRD后，无法用一句话概括产品定位
- **为什么重要**：没有定位，所有功能设计都没有方向
- **改进建议**：补充产品定位声明（为谁、解决什么问题、提供什么价值）
- **参考依据**：《决胜B端》第4章——产品定位是整体方案设计的核心

### R2. 核心业务流程缺失
- **触发条件**：PRD没有描述核心业务流程，或流程明显不完整
- **判断标准**：无法从PRD中还原出业务从开始到结束的完整路径
- **为什么重要**：不理解业务流程就无法设计正确的产品方案
- **改进建议**：用跨部门泳道图绘制核心业务流程
- **参考依据**：《决胜B端》2.1节——B端产品建设第一步是梳理核心业务流程

### R3. 数据模型严重缺陷
- **适用范围**：**仅适用于业务型管理软件和交易型平台**；工具型软件、基础服务型产品不适用此项
- **触发条件**：PRD涉及多个业务实体但完全无ER图，或ER图中实体关系明显错误
- **判断标准**：核心业务实体缺失、实体间关系错误（如一对多关系画成了多对多）、关键实体没有属性定义
- **为什么重要**：数据模型是系统的地基，一旦定型修改成本极高
- **改进建议**：重新进行ER建模，覆盖所有核心业务实体和关系
- **参考依据**：《决胜B端》5.1节——"ER建模最终会转换成数据库表设计，一旦表结构定型，以后再想修改，是非常麻烦的事情"

### R4. 角色权限完全缺失
- **触发条件**：PRD涉及多角色但完全没有权限设计
- **判断标准**：无法判断哪个角色能做什么、能看到什么
- **为什么重要**：B端产品的核心特征之一是"目标用户是一个群体，需要支撑分工协作"
- **改进建议**：建立角色权限矩阵，至少精确到页面级别
- **参考依据**：《决胜B端PRD模板》第12节角色权限矩阵

### R5. 功能存在致命断裂
- **触发条件**：核心业务流程中存在功能断点——某个环节无功能支撑导致业务无法继续
- **判断标准**：沿核心流程走查，发现"创建了但无法查询""提交了但无法审批""发货了但无法确认收货"等断点
- **为什么重要**：功能断裂意味着用户无法完成核心业务操作
- **改进建议**：沿核心流程逐一走查，补全所有断点
- **参考依据**：B端产品设计的业务闭环原则

### R6. 需求与业务严重脱节
- **触发条件**：PRD设计的功能明显不符合实际业务需求，或基于错误的业务假设
- **判断标准**：与业务方确认后发现PRD中描述的业务流程、角色关系、数据关系与实际不符
- **为什么重要**：基于错误假设的产品方案，做得越多错得越多
- **改进建议**：重新进行业务调研，校正业务理解
- **参考依据**：《决胜B端》第3章——业务调研是产品设计的前提

### R7. 合规性风险
- **适用范围**：涉及强监管行业（金融/医疗/教育/政务等）时适用
- **触发条件**：涉及强监管行业但完全未考虑合规要求
- **判断标准**：产品设计违反了已知的法规要求（如不做数据加密、不做实名认证、不做审计追踪）
- **为什么重要**：合规违规可能导致法律风险和企业损失
- **改进建议**：梳理行业合规要求，在PRD中补充合规设计
- **参考依据**：《决胜B端》调研计划中"政策法规"维度

### R8. SaaS产品无多租户概念
- **适用范围**：**仅适用于商业化SaaS产品**；企业自研内部系统不适用
- **触发条件**：商业化SaaS产品但完全无租户隔离概念
- **判断标准**：不同租户的数据可能互相可见，或功能配置无法按租户独立管理
- **为什么重要**：数据泄露是SaaS产品的生死线
- **改进建议**：补充多租户架构设计，明确数据隔离策略
- **参考依据**：《决胜B端》SaaS产品数据安全要求

## 快速诊断流程

如果时间有限，可以按以下顺序进行快速诊断：

1. **先检查重大风险项**（R1-R8）——有触发的项明确告知，建议优先处理
2. **再检查P0级问题**——高优先级问题，建议尽快处理
3. **然后检查P1级问题**——建议上线前修订
4. **最后检查P2/P3**——作为改进建议输出

## 问题优先级说明

- **P0（高优先级）**：影响核心业务流程，或存在合规/安全风险，建议尽快处理
- **P1（建议修订）**：影响重要业务场景或用户体验，建议上线前解决
- **P2（可改进）**：影响次要场景或影响有限，按资源情况安排
- **P3（优化建议）**：能提升产品质量，有时间再做

### 七、综合建议

基于以上检查，概括说明：
- 这份PRD整体上做得好的地方是什么
- 最需要关注和改进的3-5个方向
- 如果资源有限，最先应该做什么

不做通过/不通过的判定，最终如何推进由PRD作者和团队根据实际情况决定。

## 第二部分：检查执行指南

### 2.1 检查前的准备工作

**检查人需要准备：**
1. 完整的PRD文档（最新版本）
2. 如果有相关的原型/设计稿，一并获取
3. 了解PRD的类型：新产品设计/迭代需求/模块需求
4. 了解产品属性：自研系统/商业化SaaS/项目制

**根据PRD类型调整检查侧重：**

| PRD类型 | 侧重维度 | 可降低要求的维度 |
|---------|---------|----------------|
| 新产品完整PRD | 所有维度同等重要 | 无 |
| 迭代需求PRD | 05文档结构、07数据、08流程、09交互、12异常、14运营 | 01业务（简要即可）、03定位（引用上位文档）、10商业、11MVP |
| 模块级需求PRD | 07数据、08流程、09交互、12异常 | 01业务、02类型、03定位、06架构、10商业、11MVP、14运营 |
| SaaS新产品PRD | 所有维度+强化10商业、11MVP、14运营 | 无 |
| 企业自研系统PRD | 01业务、04场景、07数据、08流程、14运营 | 03定位（竞品分析可简化）、10商业（ROI替代市场分析） |

### 2.2 检查执行流程

**标准流程（完整检查，约60-90分钟）：**

1. **快速浏览**（10分钟）
   - 通读PRD全文，获取整体印象
   - 判断PRD类型和产品属性
   - 标注第一印象中明显的问题

2. **重大风险项快速排查**（5分钟）
   - 逐一检查R1-R8风险项
   - 如有发现，记录并继续（不提前终止，完成全面检查）

3. **14维度逐一检查**（40-60分钟）
   - 按维度顺序逐一执行检查清单
   - 每个检查项标注"发现问题""未发现问题""不适用"
   - 有问题的检查项记录具体问题描述和严重程度

4. **汇总与报告**（10-15分钟）
   - 汇总各维度关键发现
   - 汇总问题清单并排优先级
   - 撰写Top 10改进建议

**快速流程（快速检查，约20-30分钟）：**

1. 检查重大风险项（R1-R8）
2. 检查P0级问题对照表
3. 检查P1级问题对照表
4. 快速浏览P2/P3问题对照表
5. 输出简要检查结论

### 2.3 检查执行指导原则

1. **按PRD类型调整预期**：迭代需求PRD不需要完整的商业分析，但仍然需要清晰的需求背景和目标
2. **看整体不纠结个别**：如果某个维度整体质量高但个别检查项缺失，不必特别强调
3. **区分"缺失"和"不适用"**：某些检查项对特定产品不适用（如非SaaS产品不需要多租户设计），应标注"不适用"
4. **亮点也要记录**：检查不只是找问题，优秀的设计也值得记录和学习
5. **建议要可操作**：改进建议应该具体、可执行，而非笼统的方向性建议

### 2.4 检查结果的沟通

**沟通原则：**
1. **先说亮点再说问题**：建立积极的沟通氛围
2. **问题分级呈现**：按P0-P3分级，让PRD作者知道什么最紧急
3. **提供改进方向而非批评**：每个问题都附上参考依据和改进建议
4. **区分"必须改"和"建议改"**：给PRD作者明确的行动指引
5. **可以引用方法论**：引用《决胜B端》《决胜体验设计》中的方法论帮助PRD作者理解为什么这是个问题

**检查报告的传递：**
- P0和否决项问题：当面沟通 + 书面记录
- P1问题：书面记录 + 约定修复时间
- P2/P3问题：书面记录，PRD作者自行安排改进
